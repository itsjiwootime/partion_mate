# E2-1 Store Query Baseline

## 측정 조건
- 실행 일시: 2026-03-15
- 실행 명령: `./scripts/run_store_query_baseline.sh`
- 측정 대상:
  - `GET /api/stores/nearby`
  - `GET /api/stores/{storeId}/parties`
- 프로필: `test`
- 실행 방식: `MockMvc`
- DB: H2 in-memory
- 데이터셋:
  - 지점 300개
  - 지점당 파티 6개
  - 총 파티 1800개
- 반복 횟수:
  - 워밍업 5회
  - 측정 20회

## 측정 결과
- 주변 지점 조회 API
  - 평균 응답시간: `71.78ms`
  - P95 응답시간: `81.07ms`
  - 응답 지점 수: `300`
- 지점별 파티 조회 API
  - 평균 응답시간: `1.78ms`
  - P95 응답시간: `2.69ms`
  - 응답 파티 수: `6`

## 해석
- 현재 주변 지점 조회는 `storeRepository.findAll()` 이후 애플리케이션 메모리에서 거리 계산과 정렬을 수행하고, 지점마다 모집 중 파티 수를 별도 조회한다.
- 같은 데이터셋 기준으로 지점별 파티 조회는 매우 가볍지만, 주변 지점 조회가 Epic 2 최적화의 핵심 병목 지점임을 확인했다.
- 현재 기준 수치는 E2-2 이후 거리 계산 쿼리화와 E2-3 인덱스 작업의 비교 기준으로 사용한다.

## 재실행 메모
- 이 문서의 수치는 `Epic 2 최적화 이전 상태`에서 기록한 H2 + MockMvc 기준 기준값이다.
- 현재 브랜치에서 같은 테스트 클래스를 다시 실행하면 E2-2 이후 쿼리 최적화와 E2-4 캐시가 반영된 값이 출력된다.
- 이후 MySQL/Docker 환경 측정이 추가되면 별도 문서로 비교한다.
