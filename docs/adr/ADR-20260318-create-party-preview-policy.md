# ADR-20260318-create-party-preview-policy

- 날짜: 2026-03-18
- 상태: Accepted
- 관련 작업: `E17-2`
- 관련 파일: `frontend/src/pages/CreateParty.jsx`, `frontend/src/utils/createPartyPreview.js`, `frontend/src/utils/createPartyPreview.test.js`

## 배경
- 파티 생성 화면은 단계화되었지만, 총 가격, 총 수량, 호스트 수량, 최소 소분 단위의 관계를 사용자가 직접 계산해야 했다.
- 특히 호스트가 많이 가져가거나 최소 소분 단위가 큰 경우 참여자가 실제로 들어오기 어려운 조건이 생기는데, 생성 시점에는 이런 위험이 바로 드러나지 않았다.
- 정산 로직은 총 금액을 수량 기준으로 나누고 나머지 1원을 큰 수량 요청자부터 배분하는 방식이라, 화면에서 단순 반올림만 보여주면 실제 기대 금액과 어긋날 수 있었다.

## 결정사항
- 생성 화면에 순수 계산 유틸을 두고, `총 가격`, `총 수량`, `호스트 수량`, `최소 소분 단위`만으로 예상 범위와 경고를 계산한다.
- 금액은 단일 값 대신 `최소 ~ 최대` 범위로 노출해 1원 단위 나머지 배분 가능성을 그대로 드러낸다.
- 경고는 저장 차단이 아니라 정보 제공으로만 처리한다. 실제 생성 차단 규칙은 기존 폼 검증 범위에만 남긴다.
- 최종 확인 단계에도 같은 미리보기 값을 재사용해, 사용자가 마지막 제출 직전까지 같은 기준을 보게 한다.

## 대안
- 평균 금액 하나만 보여주는 방식
  - 장점: UI가 단순하고 빠르게 이해된다.
  - 단점: 1원 단위 나머지 배분 규칙이 사라져 실제 정산 기대치와 어긋날 수 있다.
- 백엔드에서 미리보기 계산 API를 별도로 제공하는 방식
  - 장점: 정산 정책 변경 시 서버 기준으로 단일 계산식을 유지할 수 있다.
  - 단점: 생성 폼 입력 중 매번 네트워크 왕복이 필요하고, 현재 범위에서는 프론트 순수 계산으로 충분하다.
- 경고 대신 강제 차단 규칙을 추가하는 방식
  - 장점: 모집이 어려운 조건을 사전에 더 강하게 막을 수 있다.
  - 단점: 현재 도메인은 해당 조건을 금지하지 않으므로 UI만 더 엄격해져 실제 정책과 어긋난다.

## 결과
- 긍정적 트레이드오프: 사용자가 생성 중 참여 조건을 더 빠르게 이해할 수 있고, 금액과 수량 관계를 마지막 제출 전까지 일관되게 확인할 수 있다.
- 부정적 트레이드오프: 범위형 금액과 경고 문구가 추가되어 생성 화면이 이전보다 더 많은 정보를 담게 된다.
- 중립적 트레이드오프 또는 후속 영향 사항: 이 미리보기는 예상치이므로, 향후 정산 정책이 바뀌면 `createPartyPreview` 유틸과 문서를 함께 갱신해야 한다.
