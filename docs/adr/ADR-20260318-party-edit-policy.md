# ADR-20260318-party-edit-policy

- 날짜: 2026-03-18
- 상태: Accepted
- 관련 작업: `E13-1`
- 관련 파일: `src/main/java/com/project/partition_mate/service/PartyService.java`, `src/main/java/com/project/partition_mate/domain/Party.java`, `src/main/java/com/project/partition_mate/controller/PartyController.java`

## 배경
- 파티 생성 이후에는 호스트가 제목, 수량, 마감시간, 소분 조건 같은 핵심 운영 정보를 직접 고칠 수 있는 API가 없었다.
- 이 상태에서는 오입력이나 운영 조건 변경이 생겨도 파티를 새로 만들거나 비공식 채널로만 공지해야 해서 사용성과 운영 일관성이 떨어졌다.
- 반대로 정산이나 픽업이 시작된 뒤에도 거래 조건을 자유롭게 바꾸면 참여자 금액, 약속 시간, 신뢰도 판단 기준이 흔들릴 수 있다.

## 결정사항
- 파티 수정은 `PUT /party/{id}` 단일 API로 제공하고, 호스트만 호출할 수 있게 한다.
- 수정 가능 필드는 `title`, `productName`, `totalPrice`, `totalQuantity`, `deadline`, `unitLabel`, `minimumShareUnit`, `storageType`, `packagingType`, `hostProvidesPackaging`, `onSiteSplit`, `guideNote`로 제한한다.
- `storeId`와 `hostRequestedQuantity`는 수정 대상에서 제외한다. 지점 변경은 조회/캐시/참여 맥락을 크게 흔들고, 호스트 수량 변경은 정산 및 참여 수량 모델과 강하게 엮여 있기 때문이다.
- `CLOSED` 파티나 마감 시간이 지난 파티는 수정할 수 없게 한다.
- 정산이 확정되었거나 픽업 일정이 잡힌 뒤에는 거래 조건 변경을 막고, `guideNote`만 수정 가능하게 한다.
- 수정 시 현재 참여 수량보다 `totalQuantity`를 작게 줄일 수 없게 하고, 수정 후에는 파티 상태를 다시 계산해 `FULL -> RECRUITING` 복귀도 허용한다.
- 수정이 완료되면 지점 조회 캐시를 무효화해 목록/상세의 후속 조회가 새 값을 읽도록 한다.

## 대안
- 수정 API 없이 호스트 취소 후 재생성을 강제한다.
  - 장점: 상태 전이와 예외 케이스가 단순하다.
  - 단점: 기존 참여자, 대기열, 채팅 맥락이 끊기고 실사용성이 크게 떨어진다.
- `PATCH` 기반 부분 수정 API로 모든 필드를 선택적으로 받는다.
  - 장점: 프론트가 바뀐 필드만 보내면 되어 유연하다.
  - 단점: 필드별 기본값, null 의미, 상태 제한 비교 로직이 더 복잡해지고 현재 DTO 스타일과도 거리가 있다.
- 정산/픽업 이후에도 모든 필드 수정을 허용한다.
  - 장점: 호스트 운영 자유도가 높다.
  - 단점: 금액, 약속, 후기 판단 기준이 사후 변경될 수 있어 분쟁 가능성이 커진다.

## 결과
- 긍정적 트레이드오프를 작성한다.
  - 호스트가 파티를 다시 만들지 않고도 핵심 입력 오류와 모집 조건을 수정할 수 있다.
  - 정산/픽업 이후에는 거래 조건을 잠가 참여자 관점의 예측 가능성을 지킬 수 있다.
  - 상태 재계산과 캐시 무효화까지 묶어 수정 직후 후속 조회 일관성을 유지할 수 있다.
- 부정적 트레이드오프를 작성한다.
  - `PUT` 전체 수정 방식이라 프론트가 수정 폼에서 현재 값을 모두 채워 보내야 한다.
  - `hostRequestedQuantity`나 지점 이동 같은 더 복잡한 수정 요구는 별도 카드로 나눠야 한다.
  - 실시간 전파와 채팅 시스템 메시지 동기화는 이번 카드 범위에서 제외되어 다음 카드에서 보강이 필요하다.
- 중립적 트레이드오프 또는 후속 영향 사항을 작성한다.
  - 이후 `E13-3`에서 SSE, 채팅 시스템 메시지, 알림까지 연결하면 수정 이벤트의 사용자 체감 일관성이 완성된다.
- 안전하게 열어둔 `guideNote` 수정 가능 범위는 운영 피드백에 따라 더 좁히거나 넓힐 수 있다.
