# ADR-20260315 Party Participation Schema

## 배경
파티 참여자는 `(party_id, user_id)` 기준으로 한 번만 저장되어야 하고, 정원 초과 시 대기열로 전환할 다음 카드들을 위해 별도 대기열 저장 구조가 필요했다. 동시에 대기열 이력은 향후 승격, 만료, 취소 상태 전이를 표현할 수 있어야 했다.

## 결정사항
`partymember`에는 `(party_id, user_id)` 유니크 제약을 유지하고, 신규 `waiting_queue` 테이블은 `WaitingQueueEntry` 엔티티로 분리했다. 대기열 항목은 `requestedQuantity`, `status`, `queuedAt`을 가지며, 동일 파티에서 동일 사용자에 대한 같은 상태의 중복 저장을 막기 위해 `(party_id, user_id, status)` 유니크 제약을 적용했다. 이를 통해 현재 활성 대기열 중복을 막으면서도 이후 `WAITING -> PROMOTED/CANCELED/EXPIRED` 상태 변경 뒤 재등록 여지는 남긴다.

## 대안
대안 1은 대기열 없이 `party_member` 하나의 테이블만 두고 정원 초과 시 실패 처리하는 방식이었다. 구현은 단순하지만, 이후 대기열 승격과 만료 이력을 표현하기 어렵고 E1-3 이후 요구사항을 다시 뒤엎어야 한다는 단점이 있다.

대안 2는 `waiting_queue`에 `(party_id, user_id)`만 유니크 제약으로 두는 방식이었다. 활성 중복 차단은 쉽지만, 취소나 만료 이후 동일 사용자가 같은 파티에 다시 대기 등록해야 하는 경우 기존 행을 재사용해야 해서 상태 전이와 이력 관리가 더 경직된다.

대안 3은 대기열 이력을 별도 히스토리 테이블로 나누는 방식이었다. 이력 표현은 가장 명확하지만, 현재 단계에서는 테이블과 전이 규칙이 과도하게 늘어나 구현 복잡도가 커진다.

## 결과
긍정적으로는 참여자와 대기자를 분리한 스키마가 생겨 이후 대기열 등록, 승격, 만료 카드를 자연스럽게 이어갈 수 있다. 또한 저장소 레벨에서 중복 저장을 즉시 검증할 수 있어 테스트 가능성이 좋아진다.

부정적으로는 `waiting_queue`에 상태 컬럼과 유니크 제약을 함께 둔 만큼 상태 전이 규칙을 명확히 지키지 않으면 이후 재등록 시나리오에서 예외가 날 수 있다. 향후 승격/취소 구현 시 상태 변경과 재등록 순서를 더 신중하게 다뤄야 한다.

중립적으로는 참여자와 대기자 간 중복 금지는 아직 단일 DB 제약으로 강제하지 않고 서비스 규칙으로 풀어야 한다. 현재 결정은 다음 카드에서 서비스 계층 검증을 얹는 전제를 포함한다.
