# ADR-20260315 Party Join Benchmark Method

## 배경
E1-5의 목표는 이력서에 적을 수 있는 정량 수치를 남기되, 현재 저장소 안에서 반복 실행 가능한 방식이어야 한다는 점이다. 별도 부하 도구를 당장 설치하지 않아도 재현 가능한 측정 기준과 한계를 문서화할 필요가 있었다.

## 결정사항
1차 벤치마크는 `scripts/run_party_join_load_benchmark.sh`가 실행하는 `PartyJoinLoadBenchmarkTest`로 측정한다. 측정 환경은 `test` 프로필의 H2 DB와 서비스 계층 호출이며, 동시 참여 100건 요청에서 `joined`, `waiting`, 평균 응답시간, P95를 기록한다. 초과 모집 0건과 대기열 건수도 함께 검증한다.

## 대안
대안 1은 k6 같은 외부 HTTP 부하 도구를 바로 도입하는 방식이었다. HTTP 레벨에 더 가깝다는 장점이 있지만, 현재 프로젝트에는 매장 시드와 인증 토큰 발급을 포함한 벤치마크 준비 경로가 아직 부족해 초기 비용이 크다.

대안 2는 단순 통합 테스트만 두고 성능 수치는 문서화하지 않는 방식이었다. 회귀 검증은 되지만, 이력서와 로드맵 목적에 필요한 평균/P95 수치를 남길 수 없다.

대안 3은 실제 MySQL과 운영 유사 인프라를 띄운 뒤 측정하는 방식이었다. 결과의 현실성은 높지만 현재 단계에서는 Docker 구성, 시드, 측정 자동화까지 한 번에 커져 카드 범위를 넘어선다.

## 결과
긍정적으로는 저장소 안에서 바로 실행 가능한 측정 루프가 생겨 구현 이후 수치를 반복 재생산할 수 있다. 또한 초과 모집 0건, 대기열 건수, 평균/P95를 한 번에 검증해 품질과 성능을 함께 볼 수 있다.

부정적으로는 서비스 계층 + H2 기반이라 실제 HTTP 오버헤드와 MySQL 락 비용을 반영하지 못한다. 따라서 이 수치는 1차 내부 기준값으로만 해석해야 하고, 이후 실제 인프라 측정으로 보완해야 한다.

중립적으로는 외부 도구 대신 테스트 코드를 택했기 때문에 CI에 편입하기 쉽지만, 반대로 성능 수치가 실행 환경 영향을 더 크게 받을 수 있다. 문서에는 항상 측정 조건을 함께 남겨야 한다.
