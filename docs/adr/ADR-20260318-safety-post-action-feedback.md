# ADR-20260318 Safety Post Action Feedback

## 배경
- 신고와 차단 진입점은 추가되었지만, 액션 후 화면이 계속 이전 상태를 유지하거나 단순 토스트만 보여주면 사용자가 무엇이 달라졌는지 알기 어려웠다.
- 특히 차단은 즉시 상호작용 제한으로 이어지므로, 해당 화면에서 바로 제한 상태를 설명하는 fallback이 필요했다.

## 결정사항
- 신고 성공 후에는 현재 화면에 상태 배너를 띄우고, 프로필의 `신뢰·안전 관리` 카드로 이동하는 CTA를 제공한다.
- 차단 성공 후에는 현재 화면을 그대로 두지 않고 제한 상태 fallback 또는 제한 배너를 보여준다.
- 차단 접근이 서버에서 `FORBIDDEN`으로 반환되면 일반 오류 메시지 대신 안전 fallback을 우선 노출한다.
- 프로필은 다른 화면에서 전달한 `focusSafetyCenter`, `safetyNotice` state를 읽어 차단 해제와 신고 내역 확인의 도착점 역할을 한다.

## 대안
- 토스트만 유지하는 방안
  - 장점: 구현이 가장 단순하다.
  - 단점: 액션 이후 화면 변화가 없어 사용자가 실제 제한 여부와 다음 행동을 이해하기 어렵다.
- 신고/차단 직후 무조건 프로필로 리다이렉트하는 방안
  - 장점: 관리 화면으로 바로 이동시켜 후속 행동이 명확하다.
  - 단점: 사용자가 현재 문맥을 잃고, 왜 이동됐는지 혼란스러울 수 있다.
- 차단 후에도 기존 화면을 유지하고 버튼만 비활성화하는 방안
  - 장점: 변화 폭이 작다.
  - 단점: 제한 이유와 해제 경로가 충분히 드러나지 않는다.

## 결과
- 긍정적: 신고와 차단 이후 사용자가 현재 상태와 다음 행동을 한 화면에서 바로 이해할 수 있다.
- 부정적: 화면별 상태 관리가 늘어나고, 차단 제한 메시지 판단 로직이 프론트에 일부 추가된다.
- 중립적: 현재는 프론트에서 메시지 기반으로 차단 제한을 식별하며, 추후 에러 코드가 정교해지면 같은 UI를 더 안정적으로 유지할 수 있다.
