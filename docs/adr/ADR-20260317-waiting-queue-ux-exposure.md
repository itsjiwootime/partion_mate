# ADR-20260317-waiting-queue-ux-exposure

- 날짜: 2026-03-17
- 상태: Superseded
- 관련 작업: `E9-1`
- 관련 파일: `frontend/src/pages/PartyDetail.jsx`, `frontend/src/pages/JoinParty.jsx`
- 대체 ADR: `ADR-20260327-remove-waiting-queue-join-fail-fast.md`

## 배경
- 백엔드는 모집 완료 상태에서 즉시 참여 대신 대기열 등록을 지원하지만, 프론트는 `FULL` 또는 잔여 수량 0인 파티에서 참여 CTA를 숨기고 있었다.
- 사용자는 같은 파티가 상세에서는 "마감"처럼 보이는데 실제로는 대기 등록이 가능한 상태인지 구분하기 어려웠다.
- 종료된 파티와 대기열 등록 가능한 파티를 같은 비활성 상태로 다루면 백엔드 기능이 UX에서 사실상 사라진다.

## 결정사항
- 파티 상세에서는 즉시 참여가 가능할 때 `참여하기`, 즉시 참여는 불가하지만 파티가 종료되지 않았을 때 `대기열 등록하기` CTA를 따로 노출한다.
- 참여 화면에서는 `queueOnly` 상태를 계산해 제목, 설명, 버튼 라벨을 `대기열 등록` 중심으로 바꾼다.
- 참여 수량 입력은 기존과 같은 폼을 유지하되, 최소 소분 단위 이상인지와 최대 수량 규칙은 그대로 적용한다. 배수 규칙은 이 단계에서 강제하지 않는다.
- 종료 상태(`CLOSED`)만 진입 차단하고, `FULL` 상태는 대기열 등록 가능 상태로 취급한다.

## 대안
- 상세 화면에서 계속 CTA를 숨기고 참여 화면에서만 대기 등록 허용
  - 장점: 상세 화면이 단순하다.
  - 단점: 사용자가 기능 존재를 알기 어렵고 진입률이 떨어진다.
- `FULL` 상태를 모두 참여 불가로 고정
  - 장점: 상태 해석이 단순하다.
  - 단점: 이미 구현된 대기열 기능과 도메인 정책을 프론트가 무시하게 된다.
- 별도 대기열 전용 화면을 추가
  - 장점: 설명과 순번 정보를 더 자세히 보여줄 수 있다.
  - 단점: 현재 프로젝트 규모 대비 화면과 흐름이 과하게 늘어난다.

## 결과
- 긍정적
  - 모집 완료와 거래 종료를 사용자가 구분할 수 있게 됐다.
  - 기존 백엔드 대기열 기능을 새로운 API 변경 없이 그대로 활용할 수 있다.
  - 상세 화면과 참여 화면의 상태 해석이 일관된다.
- 부정적
  - 상태 계산 분기가 늘어 상세/참여 화면의 조건식이 조금 복잡해졌다.
  - 대기열 순번 자체는 참여 결과 이후 내 파티 화면에서 확인하는 구조라 즉시 상세에서 모두 보이지는 않는다.
- 중립적
  - 이후 대기열 규모가 커지면 예상 대기 인원이나 승격 예측 같은 추가 정보 노출을 다시 검토할 수 있다.
