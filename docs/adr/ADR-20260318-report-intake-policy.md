# ADR-20260318-report-intake-policy

- 날짜: 2026-03-18
- 상태: Accepted
- 관련 작업: `E14-1`
- 관련 파일: `src/main/java/com/project/partition_mate/domain/Report.java`, `src/main/java/com/project/partition_mate/service/ReportService.java`, `src/main/java/com/project/partition_mate/controller/ReportController.java`, `docs/specs/E14-report-intake.md`

## 배경
- 현재 서비스에는 후기와 신뢰 점수는 있지만, 실제 문제 상황을 운영에 전달할 신고 접수 모델이 없었다.
- 신고가 없으면 노쇼, 정산 문제, 부적절한 채팅 같은 사건이 발생해도 이후 차단, 제재, 관리자 검토 기능으로 연결할 기준 데이터가 남지 않는다.
- 반대로 너무 넓은 범위의 신고를 초기에 허용하면 무관한 제3자 신고, 중복 접수, 메시지 단위 상세 모델까지 한 번에 풀어야 해서 구현 복잡도가 커진다.

## 결정사항
- 신고 엔티티는 `reporter`, `party`, `targetUser`, `targetType`, `reasonType`, `status`, `memo`, `createdAt`만 갖는 최소 모델로 시작한다.
- 신고 대상 타입은 `PARTY`, `USER`, `CHAT` 세 가지로 고정한다.
- 모든 사용자 신고와 채팅 신고는 반드시 `partyId`를 동반하게 해, 파티 맥락 없는 무관한 사용자 신고는 초기에 막는다.
- 파티 신고는 참여자 또는 대기 사용자만 접수할 수 있게 하고, 채팅 신고는 신고자와 대상자 모두 해당 파티 참여자일 때만 허용한다.
- 중복 정책은 `같은 신고자 + 같은 대상 타입 + 같은 파티 + 같은 대상 사용자 + 같은 사유 + RECEIVED 상태` 기준으로 막는다.
- 최소 조회 범위는 `GET /api/reports/me`로 한정해, 사용자가 본인이 접수한 신고만 최신순으로 확인할 수 있게 한다.

## 대안
- 파티 맥락 없이 모든 사용자 ID에 대한 신고를 허용한다.
  - 장점: 사용자 신고 API가 단순해 보인다.
  - 단점: 실제 거래 관계가 없는 제3자 신고를 막기 어렵고, 악용 방지 정책이 더 빨리 필요해진다.
- 채팅 신고를 메시지 ID 단위로 바로 설계한다.
  - 장점: 관리자 검토 시 어떤 메시지를 문제 삼았는지 더 정확하게 볼 수 있다.
  - 단점: 현재 채팅 API와 화면에는 메시지 고유 ID 기반 신고 흐름이 없어서 이번 카드 범위를 크게 넘긴다.
- 신고 생성만 두고 조회 API는 나중으로 미룬다.
  - 장점: 백엔드 구현량이 줄어든다.
  - 단점: 사용자가 신고가 실제 접수됐는지 확인할 수 없고, 추후 프론트 안전 UX 연결점도 사라진다.

## 결과
- 긍정적
  - 노쇼, 정산 문제, 채팅 문제를 운영 데이터로 남길 최소 구조가 생겼다.
  - 파티 맥락을 강제해 신고 남용 가능성을 어느 정도 줄였다.
  - 이후 차단, 관리자 검토, 제재, 통계 기능이 같은 신고 ID를 기준으로 확장될 수 있다.
- 부정적
  - 아직 관리자 처리 상태가 `RECEIVED` 한 단계뿐이라 후속 조치 이력은 저장하지 않는다.
  - 채팅 신고가 메시지 단위가 아니어서 정밀한 증거 보존에는 한계가 있다.
- 중립적
  - 이후 운영 도구가 생기면 `REVIEWING`, `RESOLVED`, `REJECTED` 같은 상태 확장과 담당자 메모 필드가 필요해질 수 있다.
  - 프론트 안전 UX를 붙일 때는 현재 `GET /api/reports/me` 응답을 그대로 사용하고, 관리자용 조회는 별도 엔드포인트로 분리하는 편이 낫다.
