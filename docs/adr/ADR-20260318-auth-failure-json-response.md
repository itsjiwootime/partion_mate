# ADR-20260318 인증 실패 JSON 응답 표준화

## 배경
로그인 만료나 잘못된 토큰이 들어왔을 때 현재 서비스는 Spring Security 기본 동작에 기대고 있었다. 그 결과 보호 API가 `403`과 빈 본문으로 끝나는 경우가 있었고, 프론트는 세션 만료와 일반 권한 거부를 구분하지 못했다. 리프레시 토큰 도입 전에 인증 실패 응답을 먼저 표준화해야 프론트가 access token 만료를 감지하고 재로그인 또는 재발급 흐름으로 연결할 수 있다.

## 결정사항
Spring Security에 전용 `AuthenticationEntryPoint`와 `AccessDeniedHandler`를 추가해 인증 실패 응답을 모두 JSON으로 통일한다. JWT 필터는 만료 토큰과 변조/잘못된 토큰을 구분해 request attribute로 에러 코드와 메시지를 기록하고, 엔트리포인트가 이를 읽어 `AUTH_REQUIRED`, `TOKEN_EXPIRED`, `INVALID_TOKEN`, `ACCESS_DENIED` 코드로 응답한다. 인증되지 않은 요청과 잘못된 토큰은 `401`, 인증은 되었지만 권한이 없는 경우는 `403`으로 구분한다.

## 대안
기본 Spring Security 응답을 그대로 사용한다.
- 장점: 구현이 가장 단순하고 추가 코드가 거의 없다.
- 단점: 기본 응답이 빈 본문 또는 HTML 에러로 끝날 수 있어 SPA가 세션 만료를 일관되게 처리하기 어렵다.

JWT 필터에서 직접 응답을 작성하고 체인을 중단한다.
- 장점: 필터 한 곳에서 토큰 예외를 즉시 처리할 수 있다.
- 단점: 인증 실패 응답과 권한 거부 응답이 분리되지 않고, 시큐리티 표준 흐름과 중복 로직이 생긴다.

컨트롤러 예외 처리기만 사용한다.
- 장점: 기존 `GlobalExceptionHandler`와 응답 포맷을 공유하기 쉽다.
- 단점: 시큐리티 필터 체인에서 발생하는 인증 실패는 컨트롤러까지 도달하지 않아 적용 범위가 불완전하다.

## 결과
긍정적 측면:
- 프론트가 세션 만료와 인증 필요 상태를 코드 기반으로 구분할 수 있다.
- 리프레시 토큰 도입 전 단계에서 인증 실패 UX를 안정화할 수 있다.
- 보안 오류 응답 형식이 API 전체에서 일관된다.

부정적 측면:
- 시큐리티 전용 클래스와 테스트가 추가되어 인증 계층 복잡도가 약간 증가한다.
- request attribute를 통한 에러 전달 규약을 필터와 엔트리포인트가 함께 알아야 한다.

중립적 측면:
- 현재는 역할 기반 권한 제한 엔드포인트가 많지 않아 `403` 경로 활용도는 낮지만, 이후 관리자/운영 기능 추가 시 그대로 재사용할 수 있다.
