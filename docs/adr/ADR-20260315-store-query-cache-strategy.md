# ADR-20260315-store-query-cache-strategy

## 배경
- 홈 화면과 파티 생성 화면은 같은 좌표 기준 주변 지점 조회를 반복 호출하고, 지점 상세 흐름에서는 같은 `storeId`로 파티 목록을 자주 다시 읽는다.
- 조회 쿼리 최적화만으로 cold path는 충분히 줄였지만, 반복 호출까지 고려하면 캐시 계층이 있어야 Epic 2 성능 목표를 안정적으로 달성할 수 있었다.
- 다만 현재 저장소에는 Redis 인프라가 준비되어 있지 않아, 로컬 테스트와 GitHub Actions 수준에서도 재현 가능한 방식이 필요했다.

## 결정사항
- Epic 2 단계에서는 `Spring Cache + Caffeine`으로 in-process 캐시 계층을 먼저 도입한다.
- 캐시 키는 주변 지점 조회는 `위도:경도(소수점 4자리)` 문자열, 지점별 파티 조회는 `storeId`를 사용한다.
- TTL은 주변 지점 조회 5분, 지점별 파티 조회 2분으로 설정한다.
- 무효화 규칙은 `파티 생성`, `즉시 참여`, `참여 취소/승격` 시 `nearbyStores` 전체와 해당 `storeId`의 `storeParties`를 비운다.

## 대안
- Redis를 즉시 도입
  - 장점: 멀티 인스턴스 환경까지 고려한 캐시 구성이 가능하다.
  - 단점: 현재 프로젝트에는 Redis 실행 환경과 운영 설정이 없어 Epic 2 범위 대비 준비 비용이 크다.
- 캐시 없이 쿼리 최적화만 적용
  - 장점: 구조가 단순하고 일관성 문제가 줄어든다.
  - 단점: 같은 좌표/같은 지점 재조회가 잦은 UI 흐름에서 응답시간을 더 줄일 여지가 남는다.
- 좌표 원본값 그대로를 캐시 키로 사용
  - 장점: 구현이 가장 단순하다.
  - 단점: 미세한 좌표 차이에도 캐시가 분산돼 hit ratio가 낮아질 수 있다.

## 결과
- 긍정적
  - 별도 인프라 없이도 캐시 hit/miss와 무효화 규칙을 테스트와 벤치마크로 검증할 수 있게 되었다.
  - 같은 좌표/같은 지점 재조회에서 warm path 성능을 크게 줄일 수 있었다.
  - Redis 전환 전에도 Spring Cache 추상화를 유지해 후속 전환 비용을 줄였다.
- 부정적
  - 현재 캐시는 단일 인스턴스 메모리 캐시라 멀티 인스턴스 환경에서는 공유되지 않는다.
  - 좌표를 소수점 4자리로 정규화했기 때문에 매우 촘촘한 위치 차이는 같은 캐시 키로 묶일 수 있다.
- 중립적
  - Epic 2 단계에서는 재현성과 단순성을 우선했기 때문에, 이후 CD/AWS 단계에서 Redis로 전환할 여지는 그대로 남겨두었다.
