# ADR-20260317-chat-ui-and-openchat-fallback

- 날짜: 2026-03-17
- 상태: Accepted
- 관련 작업: `E8-5`
- 관련 파일: `frontend/src/pages/Chat.jsx`, `frontend/src/pages/PartyDetail.jsx`, `frontend/src/pages/MyParties.jsx`, `frontend/src/pages/PartyList.jsx`, `frontend/src/utils/chatClient.js`

## 배경
- 기존 파티 상세에는 카카오 오픈채팅 링크가 이미 있었고, 에픽 8 이후에도 외부 채널을 바로 없애면 운영 중인 파티 흐름이 끊길 수 있다.
- 사용자는 파티 상세, 내 파티, 목록 어디서 들어오더라도 채팅 unread를 볼 수 있어야 하고, 인앱 채팅 진입 동선도 자연스러워야 한다.
- 프론트 테스트 러너가 아직 없어, 화면 통합 범위는 빌드 검증과 실제 API 계약에 맞춘 구조가 중요했다.

## 결정사항
- 채팅 화면은 `/chat`, `/chat/:partyId` 두 라우트로 제공한다.
- 하단 네비게이션에 채팅 탭을 추가하고, 파티 상세와 내 파티 화면에는 해당 파티 채팅방으로 이동하는 CTA를 둔다.
- 인앱 채팅이 기본 경로지만, `openChatUrl`이 있는 경우 채팅 화면과 파티 상세 모두 외부 오픈채팅 열기 fallback을 유지한다.
- 파티 목록과 내 파티는 `GET /api/chat/rooms` 결과를 활용해 unread count를 덧입히고, 채팅 화면에서는 REST 상세 + STOMP 구독을 함께 사용한다.

## 대안
- 인앱 채팅 도입과 동시에 오픈채팅 링크 제거
  - 장점: 사용자 흐름이 단순해진다.
  - 단점: 기존 파티 운영 습관과 예외 상황 대응 수단을 한 번에 잃게 된다.
- 채팅 화면 없이 파티 상세 내부에만 채팅 컴포넌트 삽입
  - 장점: 진입 동선이 줄어든다.
  - 단점: 채팅방 목록, unread 확인, 여러 파티 전환 UX가 나빠진다.
- unread를 채팅 탭에서만 노출
  - 장점: 구현이 단순하다.
  - 단점: 목록/내 파티에서 채팅 유입을 놓치기 쉽다.

## 결과
- 긍정적
  - 기존 오픈채팅 흐름을 끊지 않으면서 인앱 채팅을 기본 UX로 추가할 수 있게 됐다.
  - 목록, 상세, 내 파티, 전용 채팅 화면이 같은 unread/공지 모델을 공유한다.
  - `@stomp/stompjs` 재연결과 REST 재조회 조합으로 기본적인 연결 회복성을 확보했다.
- 부정적
  - 현재는 채팅방 목록과 unread가 모든 방에서 실시간 동기화되지는 않고, 방별 상세 중심으로 동작한다.
  - 프론트엔드 자동화 테스트가 없어 화면 통합은 `npm run build`와 수동 흐름 전제를 함께 가진다.
- 중립적
  - 이후 인앱 채팅 사용률이 충분히 올라가면 오픈채팅 fallback 제거 여부를 다시 판단할 수 있다.
