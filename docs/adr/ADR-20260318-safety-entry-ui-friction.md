# ADR-20260318 Safety Entry UI Friction

## 배경
- 신고와 차단 API는 이미 구현되어 있었지만, 사용자가 실제로 문제 상황에서 기능을 찾기 쉬운 UI 진입점은 부족했다.
- 신고는 상황이 발생한 화면에서 바로 접수되어야 하고, 차단 상태와 신고 내역은 별도 관리 화면에서 다시 확인할 수 있어야 했다.

## 결정사항
- 신고와 차단의 실행 진입점은 `파티 상세`와 `채팅`에 둔다.
- 차단 상태와 신고 내역의 확인/해제 진입점은 `프로필`의 `신뢰·안전 관리` 카드에 둔다.
- 신고는 대상 타입별 사유 선택과 선택 메모를 포함한 공통 모달로 처리하고, 차단은 공통 확인 모달로 처리한다.
- 차단은 즉시 실행 가능한 액션으로 두되, 확인 모달에서 상호작용 제한 결과를 먼저 설명한다.

## 대안
- 파티 상세와 채팅에 별도 화면 전환형 신고 폼을 두는 방안
  - 장점: 단계가 명확하고 입력 공간이 넓다.
  - 단점: 문제 상황에서 현재 맥락을 벗어나야 해서 이탈이 커진다.
- 프로필에만 신고/차단 기능을 모으는 방안
  - 장점: 관리 포인트가 한 곳으로 모인다.
  - 단점: 실제 문제 상황에서 기능 발견성이 떨어지고 즉시성이 약하다.
- 차단을 확인 없이 즉시 실행하는 방안
  - 장점: 마찰이 가장 낮다.
  - 단점: 되돌릴 수는 있어도 실수로 실행할 가능성이 높다.

## 결과
- 긍정적: 사용자는 문제를 겪는 화면에서 바로 신고/차단을 실행할 수 있고, 프로필에서 차단 목록과 신고 내역을 재확인할 수 있다.
- 부정적: 여러 화면에서 공통 모달을 호출하므로 프론트 상태 관리와 테스트 범위가 늘어난다.
- 중립적: 현재는 차단 직후 상세한 후속 상태 설명보다 실행 진입점 정리에 집중했고, 차단 이후 상태 피드백은 다음 카드에서 보강한다.
