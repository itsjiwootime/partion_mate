# ADR-20260317-local-mysql-docker-compose

- 날짜: 2026-03-17
- 상태: Accepted
- 관련 작업: `E10-2`
- 관련 파일: `docker-compose.yml`, `.env.example`, `.gitignore`, `docs/agent-roadmap.md`

## 배경
- 현재 백엔드는 기본 설정상 로컬 MySQL에 연결되도록 되어 있지만, 저장소 안에는 DB를 일관되게 띄우는 실행 방법이 없었다.
- 개발자마다 로컬 MySQL 설치 상태가 다르면 백엔드 실행과 시드 데이터 확인이 바로 막히기 쉽다.
- 이미 지점 시드가 앱 시작 시 적재되도록 바뀌었기 때문에, 로컬 DB만 표준화하면 프론트와 백엔드를 빠르게 재현할 수 있다.

## 결정사항
- 로컬 개발 DB는 `docker-compose.yml`로 MySQL 8.4 컨테이너 하나를 관리한다.
- DB 연결 값은 루트 `.env`에서 읽고, 저장소에는 `.env.example`만 추적한다.
- 백엔드 기본 `application.yml`의 `DB_URL`, `DB_USERNAME`, `DB_PASSWORD` 환경변수 구조는 그대로 유지한다.
- 볼륨과 healthcheck를 추가해 재시작 후에도 데이터가 유지되고, 백엔드 기동 전에 상태 확인이 가능하게 한다.

## 대안
- README에 수동 MySQL 설치 절차만 문서화
  - 장점: Docker 파일을 관리하지 않아도 된다.
  - 단점: 개발 환경 재현성이 떨어지고 설치 편차가 크다.
- Testcontainers로만 로컬 DB 대체
  - 장점: 테스트 환경은 자동화된다.
  - 단점: 실제 앱을 수동으로 띄워 확인하는 개발 흐름을 해결하지 못한다.
- 백엔드와 프론트까지 모두 compose에 포함
  - 장점: 전부 한 명령으로 띄울 수 있다.
  - 단점: 현재는 프론트 Vite 개발 서버와 백엔드 `spring-boot:run`을 분리해 쓰는 편이 디버깅이 단순하다.

## 결과
- 긍정적
  - 로컬 DB 기동 절차가 표준화돼 새 환경에서도 바로 실행할 수 있다.
  - 환경변수 구조를 유지해 기존 백엔드 설정을 크게 바꾸지 않았다.
  - 지점 시드와 결합하면 빈 DB에서도 홈/지점/파티 생성 흐름을 즉시 확인할 수 있다.
- 부정적
  - `.env`와 Docker Desktop 의존성이 생긴다.
  - 현재는 루트 계정 하나만 사용하는 단순 구성이라 운영 환경과는 차이가 있다.
- 중립적
  - 이후 CI/CD나 배포 환경을 붙일 때는 별도 `compose.prod.yml` 또는 클라우드 매니지드 DB로 분리하는 편이 자연스럽다.
