# ADR-20260318-party-discovery-filter-scope

- 날짜: 2026-03-18
- 상태: Accepted
- 관련 작업: `E16-1`
- 관련 파일: `frontend/src/pages/Home.jsx`, `frontend/src/pages/PartyList.jsx`, `frontend/src/utils/partyDiscovery.js`, `frontend/src/components/PartyCard.jsx`

## 배경
- 현재 홈과 파티 목록은 조회된 파티를 그대로 나열하는 구조라 사용자가 특정 상품이나 조건의 파티를 빠르게 찾기 어렵다.
- 아직 상품 카탈로그나 전용 검색 API를 도입하지 않은 상태에서, 탐색 경험을 개선하려면 기존 파티 요약 응답 위에서 동작하는 가벼운 필터 계층이 필요했다.
- 동시에 홈에서 본 대표 조건이 파티 목록으로 자연스럽게 이어져야 하고, 새로고침이나 링크 공유 시 같은 탐색 상태가 유지돼야 했다.

## 결정사항
- 검색 및 필터는 백엔드 신규 API 없이 현재 `GET /party/all`, `GET /api/stores/{storeId}/parties` 응답을 정규화한 뒤 프론트에서 적용한다.
- 필터 상태는 컴포넌트 내부 상태가 아니라 URL 쿼리 `q`, `status`, `storage`, `unit`으로 관리한다.
- 홈에는 검색 입력과 대표 필터 카드만 두고, 실제 상세 필터 조합은 파티 목록 화면에서 수행한다.
- 첫 카드 범위는 검색어, 상태, 보관 방식, 소분 단위, 필터 요약, 결과 0건 빈 상태까지로 제한한다.

## 대안
- 백엔드 검색/필터 API를 먼저 추가한다.
  - 장점: 데이터가 많아져도 서버에서 정확한 필터링과 정렬, 페이지네이션을 함께 처리할 수 있다.
  - 단점: 이번 단계 목표인 탐색 UX 개선보다 백엔드 설계 범위가 커지고, 사용자 자유입력 상품 구조에서는 검색 품질도 제한적이다.
- 필터 상태를 파티 목록 컴포넌트 로컬 state로만 관리한다.
  - 장점: 구현이 단순하고 라우터 의존이 줄어든다.
  - 단점: 홈에서 조건을 넘기기 어렵고 새로고침, 뒤로가기, 링크 공유 시 상태가 유지되지 않는다.
- 검색, 필터, 정렬, 발견 섹션을 한 번에 모두 구현한다.
  - 장점: 한 번에 풍부한 탐색 UI를 제공할 수 있다.
  - 단점: 카드 범위가 커져 회귀 지점을 좁히기 어렵고, 기본 필터 정책과 정렬 정책이 섞여 문서화가 흐려진다.

## 결과
- 긍정적
  - 별도 백엔드 변경 없이 홈과 파티 목록에 즉시 사용할 수 있는 탐색 진입점을 제공한다.
  - URL 기반 상태로 홈에서 파티 목록으로 조건을 자연스럽게 넘기고, 같은 조건을 다시 재현할 수 있다.
  - 필터 규칙을 `partyDiscovery` 유틸로 분리해 이후 정렬·발견 섹션 작업과 테스트 기반을 마련한다.
- 부정적
  - 필터가 현재 브라우저에 로드된 목록에만 적용되므로 데이터 규모가 커지면 성능과 정확도 한계가 있다.
  - 검색 대상이 자유입력 텍스트라 오탈자나 상품 표현 차이에 약하다.
  - 홈과 파티 목록이 같은 필터 모델을 공유하므로 쿼리 파라미터 변경 시 두 화면을 같이 관리해야 한다.
- 중립적
  - 이번 결정은 상품 카탈로그나 서버 검색 API가 도입되기 전까지의 1차 탐색 계층이다.
  - 정렬, 발견 섹션, 관심 파티 저장은 같은 쿼리 기반 탐색 모델을 확장하는 후속 카드로 이어진다.
