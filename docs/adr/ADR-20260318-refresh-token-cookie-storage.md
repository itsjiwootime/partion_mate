# ADR-20260318 리프레시 토큰 발급과 저장 구조

## 배경
access token만 사용하는 현재 구조는 만료 시간이 지나면 사용자가 다시 로그인해야 하고, 로그아웃이나 세션 연장 정책을 세밀하게 제어하기 어렵다. 이번 단계에서는 리프레시 토큰을 도입하되, 여러 기기 로그인을 허용하고 이후 rotation/로그아웃 무효화까지 자연스럽게 확장할 수 있는 저장 구조가 필요했다. 또한 프론트 저장소에 리프레시 토큰을 직접 보관하지 않도록 `HttpOnly Cookie` 기반 구조를 먼저 정해야 했다.

## 결정사항
access token은 기존처럼 JWT로 유지하고, refresh token은 서버가 생성한 난수 기반 opaque token으로 발급한다. refresh token 원문은 응답 본문이 아닌 `HttpOnly Cookie`로만 전달하고, 서버 DB에는 SHA-256 해시만 저장한다. 저장 모델은 `RefreshToken(user, tokenHash, expiresAt, revoked, createdAt, revokedAt)`로 두고, 한 사용자가 여러 기기에서 동시에 로그인할 수 있게 토큰을 여러 개 보관한다. 로그인 응답 본문에는 access token과 access token 만료 시간만 노출한다.

## 대안
refresh token도 JWT로 발급하고 서버에는 저장하지 않는다.
- 장점: DB 조회 없이도 토큰 자체 검증이 가능하다.
- 단점: 개별 토큰 무효화와 rotation, 재사용 탐지가 어렵고 로그아웃 시 서버 제어력이 약하다.

refresh token 원문을 DB에 그대로 저장한다.
- 장점: 구현이 단순하고 재발급 시 바로 조회할 수 있다.
- 단점: DB 유출 시 토큰 원문이 그대로 노출되어 보안 위험이 크다.

한 사용자당 refresh token 1개만 유지한다.
- 장점: 구현과 무효화 정책이 단순하다.
- 단점: 여러 기기 로그인 허용 정책과 맞지 않고, 모바일/웹 동시 사용이 어렵다.

## 결과
긍정적 측면:
- 프론트는 `HttpOnly Cookie`를 통해 리프레시 토큰을 직접 다루지 않아도 된다.
- 서버는 해시 저장 구조 덕분에 토큰 원문 노출 없이 rotation, 로그아웃, 재사용 대응을 확장할 수 있다.
- 여러 기기 동시 로그인 정책을 지원할 수 있다.

부정적 측면:
- 로그인 시 토큰 발급 로직과 쿠키 설정, DB 저장이 함께 들어가 인증 계층 복잡도가 증가한다.
- 이후 refresh endpoint와 로그아웃 endpoint에서 쿠키와 DB 토큰 상태를 함께 관리해야 한다.

중립적 측면:
- 현재 단계에서는 refresh token 발급과 저장만 먼저 도입하고, 실제 재발급/rotation 소모 로직은 다음 카드에서 연결한다.
