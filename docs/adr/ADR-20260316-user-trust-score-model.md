# ADR-20260316-user-trust-score-model

- 날짜: 2026-03-16
- 상태: Accepted
- 관련 작업: `E7-2`
- 관련 파일: `src/main/java/com/project/partition_mate/service/TrustScoreService.java`

## 배경
- 후기 수만 보여주면 거래 경험이 적은 신규 사용자와 노쇼 이력이 있는 사용자를 충분히 구분하기 어렵다.
- 평점만으로는 실제 거래 이행률이 반영되지 않고, 노쇼 건수만으로는 긍정적인 후기 자산이 무시된다.
- 프로필과 파티 상세에서 바로 노출할 수 있는 단일 점수와 등급이 있어야 참여자가 비교를 빠르게 할 수 있다.

## 결정사항
- 신뢰도는 후기 평균 평점, 받은 후기 수, 완료 거래 수, 노쇼 수를 조합해 0~100점 `trustScore`로 계산한다.
- 완료 거래 수는 참여자로서 완료한 거래와 호스트로서 완료시킨 참여자 거래를 모두 포함한다.
- 노쇼 수는 사용자 본인이 참여자로 기록된 `NO_SHOW`만 카운트한다.
- 산식은 `ratingScore + completionScore + reviewBonus - noShowPenalty`로 구성하고, 결과는 0~100 범위로 clamp 한다.
- 점수 외에 `NEW`, `TOP`, `GOOD`, `NORMAL`, `CAUTION` 등급을 두고, 후기/거래 이력이 전혀 없는 경우는 `NEW`로 분리한다.

## 대안
- 대안 1: 후기 평균 평점만 노출
  - 장점: 직관적이고 구현이 가장 간단하다.
  - 단점: 리뷰 표본 수가 적을 때 왜곡이 크고, 노쇼나 거래 이행률이 반영되지 않는다.
- 대안 2: 완료 거래 수와 노쇼 수만 노출하고 점수는 만들지 않음
  - 장점: 산식 논란이 적고 계산이 단순하다.
  - 단점: 사용자가 여러 지표를 직접 해석해야 해서 프로필과 상세 화면에서 판단 비용이 커진다.
- 대안 3: 배치 집계 테이블을 따로 두고 신뢰도를 사전 계산
  - 장점: 조회 성능과 집계 확장성에 유리하다.
  - 단점: 현재 프로젝트 규모에서는 실시간 일관성보다 운영 복잡도가 커지고, 배치 실패 시 신뢰도 갱신 지연 문제가 생긴다.

## 결과
- 긍정적: 평점, 이행률, 노쇼를 함께 반영해 단일 지표만 봐도 거래 신뢰를 빠르게 판단할 수 있다.
- 긍정적: 별도 배치 없이 요청 시 계산하므로 후기 작성 직후 프로필과 상세에 바로 반영된다.
- 부정적: 산식이 정책적 결정이어서 이후 실제 사용자 반응에 따라 조정이 필요할 수 있다.
- 부정적: 조회 시 집계 쿼리가 추가되므로 사용자 프로필과 상세 응답 비용이 소폭 증가한다.
- 중립적: 점수보다 등급을 먼저 보여주는 UI와 결합하면 계산식 변경의 영향이 일부 완화된다.
- 중립적: 사용자 수가 커지면 이후 materialized view나 집계 캐시로 옮기기 쉬운 서비스 경계를 `TrustScoreService`로 분리해 두었다.
