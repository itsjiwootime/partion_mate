# ADR-20260316-trust-info-exposure-ux

- 날짜: 2026-03-16
- 상태: Accepted
- 관련 작업: `E7-3`
- 관련 파일: `frontend/src/pages/Profile.jsx`

## 배경
- 신뢰도 정보를 한 화면에 몰아넣으면 사용자가 핵심 지표를 읽기 어렵고, 파티 참여 결정에 필요한 맥락도 흐려진다.
- 프로필에서는 자기 거래 이력을 자세히 보고 싶고, 파티 상세에서는 호스트를 빠르게 판단할 수 있는 최소 정보가 더 중요하다.
- 후기 작성 흐름까지 같은 화면에 들어가므로, 상세 화면에서는 신뢰도 노출과 액션 UI의 충돌을 줄여야 했다.

## 결정사항
- 프로필 화면에는 내 신뢰도 카드와 최근 받은 후기 목록을 모두 노출한다.
- 파티 상세 화면에는 호스트의 신뢰 등급, 핵심 수치, 최근 후기 3건만 요약해서 노출한다.
- 참여자가 거래를 완료하면 같은 상세 화면 안에서 바로 호스트 후기를 작성할 수 있게 하고, 호스트도 참여자 정산 카드 안에서 후기를 남기게 한다.
- 후기 작성 완료 여부는 상세 응답에 포함된 `hasReviewedHost`, `reviewWritten`으로 노출해 중복 폼을 숨긴다.

## 대안
- 대안 1: 신뢰도 정보를 프로필에만 노출
  - 장점: 상세 화면이 단순해지고 구현 범위가 줄어든다.
  - 단점: 참여 결정을 내리기 전에 프로필 이동이 필요해져 사용자 흐름이 끊긴다.
- 대안 2: 파티 상세에 전체 후기 목록을 전부 노출
  - 장점: 정보량이 많아 판단 근거가 풍부해진다.
  - 단점: 상세 화면이 과도하게 길어지고 후기 조회 비용이 커진다.
- 대안 3: 후기 작성 화면을 별도 페이지로 분리
  - 장점: 입력 UI를 넓게 설계할 수 있고 복잡한 정책을 설명하기 쉽다.
  - 단점: 거래 완료 직후 즉시 작성 흐름이 끊기고, 참여자와 호스트 모두 액션 진입 비용이 높아진다.

## 결과
- 긍정적: 파티 상세만 봐도 호스트 신뢰도를 바로 판단할 수 있어 참여 전환에 필요한 맥락이 충분해진다.
- 긍정적: 후기 작성이 거래 상세 흐름 안에 남아 있어 별도 이동 없이 완료 후 행동을 유도할 수 있다.
- 부정적: 상세 화면의 책임이 늘어나면서 상태 조합이 많아지고 프론트 로직이 복잡해진다.
- 부정적: 후기 목록을 3건으로 제한했기 때문에 오래된 평가 맥락은 생략된다.
- 중립적: 이후 공개 프로필 페이지를 만들면 현재 프로필 카드와 후기 목록 구성을 재사용할 수 있다.
- 중립적: 후기 수가 늘어나면 별도 후기 모달이나 페이징 API를 추가해 확장할 수 있다.
