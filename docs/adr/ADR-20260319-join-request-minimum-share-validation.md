# ADR-20260319-join-request-minimum-share-validation

- 날짜: 2026-03-19
- 상태: Accepted
- 관련 작업: `E9-1`
- 관련 파일: `src/main/java/com/project/partition_mate/service/PartyService.java`, `src/test/java/com/project/partition_mate/service/PartyWaitingQueueIntegrationTest.java`

## 배경
- 파티 생성 화면과 참여 화면은 `minimumShareUnit`를 입력받고, 프론트 입력창에도 최소값 제한을 걸어 두었지만 서버는 참여 요청 수량이 이 값보다 작은지 검사하지 않았다.
- 그 결과 API를 직접 호출하면 호스트가 설정한 최소 참여 기준보다 적은 수량으로 즉시 참여하거나 대기열에 등록할 수 있었다.
- 이미 저장된 잘못된 대기열 요청이 있으면 자동 승격 시점에도 그대로 참여자로 바뀔 수 있어, 프론트와 백엔드의 규칙이 어긋난 상태가 된다.

## 결정사항
- `POST /party/{id}/join` 처리에서 요청 수량이 `minimumShareUnit` 이상인지 서버가 직접 검사한다.
- `minimumShareUnit`는 참여 하한값으로만 해석하고, 배수 규칙은 강제하지 않는다.
- 자동 승격 로직에서도 최소 참여 기준보다 작은 기존 대기 요청은 승격하지 않고 `EXPIRED`로 만료 처리한다.
- 예외 메시지는 사용자가 바로 이해할 수 있도록 `요청 수량은 최소 N개 이상이어야 합니다.` 형태로 고정한다.

## 대안
- 프론트 입력 제한만 유지하고 서버는 그대로 둔다.
  - 장점: 구현 변경이 가장 작다.
  - 단점: API 직접 호출이나 우회 입력을 막을 수 없고, 잘못된 데이터가 계속 쌓인다.
- `minimumShareUnit`를 배수 규칙으로 해석해 `requestedQuantity % minimumShareUnit == 0`까지 강제한다.
  - 장점: 소분 단위를 더 엄격하게 맞출 수 있다.
  - 단점: 현재 프론트 제출 로직과 변수명이 모두 하한값 의미에 더 가깝고, 기존 사용자 기대와 충돌할 수 있다.
- 요청 수량 검증을 `PartyMember`나 `WaitingQueueEntry` 생성자에만 둔다.
  - 장점: 도메인 객체 생성 시점에 일관되게 막을 수 있다.
  - 단점: 예외 메시지와 정책 문맥이 파티 참여 서비스에서 멀어지고, 기존 잘못된 대기열 정리 로직은 별도로 필요하다.

## 결과
- 긍정적
  - 프론트와 백엔드의 최소 참여 기준이 일치한다.
  - 우회 요청으로 최소 기준보다 적은 참여 수량이 들어오는 문제를 막을 수 있다.
  - 과거에 저장된 잘못된 대기열 요청도 자동 승격 과정에서 정리된다.
- 부정적
  - 참여 요청 실패 케이스가 하나 더 늘어나 클라이언트가 `400` 응답을 처리해야 한다.
  - `minimumShareUnit`를 배수 규칙으로 기대한 사용자가 있다면 여전히 더 느슨하다고 느낄 수 있다.
- 중립적
  - 이후 제품이 실제 배수 규칙을 원하면, 필드 이름과 프론트 UX를 함께 바꾸는 후속 작업이 필요하다.
