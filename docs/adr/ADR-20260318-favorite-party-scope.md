# ADR-20260318-favorite-party-scope

- 날짜: 2026-03-18
- 상태: Accepted
- 관련 작업: `E16-3`
- 관련 파일: `src/main/java/com/project/partition_mate/domain/FavoriteParty.java`, `src/main/java/com/project/partition_mate/service/FavoritePartyService.java`, `frontend/src/pages/FavoriteParties.jsx`

## 배경
- 파티 탐색과 정렬이 추가되면서 사용자가 나중에 다시 볼 파티를 따로 저장할 수 있는 기능이 필요해졌다.
- 현재 구조는 공개 조회 API와 인증 사용자 전용 API가 분리되어 있고, 파티 목록은 지점별 캐시를 사용하고 있다.
- 따라서 관심 파티 저장 기능은 사용자별 상태를 추가하되, 기존 공개 조회 캐시와 파티 조회 흐름을 크게 흔들지 않는 방식이 필요했다.

## 결정사항
- 관심 파티는 `favorite_party` 별도 엔티티로 분리하고 `user_id + party_id` 유니크 제약으로 중복 저장을 막는다.
- 저장/해제 API는 `PUT/DELETE /api/users/me/favorite-parties/{partyId}` 로 두고 두 요청 모두 멱등하게 처리한다.
- 공개 조회 API는 유지하되, 로그인 사용자가 호출한 경우에만 응답 DTO의 `favorite` 필드를 계산해 덧붙인다.
- 지점별 파티 조회는 캐시된 기본 응답을 그대로 두고, 사용자별 `favorite` 플래그만 후처리로 덧입힌다.
- 프론트는 목록, 상세, 내 관심 파티 페이지에서 같은 토글 동작을 사용하고, 비로그인 사용자는 로그인으로 유도한다.

## 대안
- `User` 엔티티에 관심 파티 목록을 직접 매핑한다.
  - 장점: 사용자 중심 도메인으로 보기에 단순하다.
  - 단점: 사용자 조회와 파티 조회가 더 강하게 결합되고, 저장 시점과 조회 인덱스를 세밀하게 다루기 어렵다.
- 관심 파티 전용 조회 API만 두고 공개 조회 응답에는 `favorite`를 넣지 않는다.
  - 장점: 공개 API와 인증 상태를 완전히 분리할 수 있다.
  - 단점: 목록과 상세에서 저장 상태를 다시 맞추려면 추가 호출이 필요해져 프론트 복잡도와 요청 수가 늘어난다.
- 비로그인 포함 모든 조회에서 서버 세션 없이 로컬 스토리지로만 저장한다.
  - 장점: 백엔드 변경이 적다.
  - 단점: 기기 간 동기화가 되지 않고, 웹 푸시/내 파티처럼 계정 기반 기능과 흐름이 분리된다.

## 결과
- 긍정적
  - 목록, 상세, 내 관심 파티 페이지가 같은 저장 모델을 공유해 사용 흐름이 단순해진다.
  - 공개 조회 API와 기존 캐시 구조를 유지하면서 로그인 사용자에게만 관심 상태를 노출할 수 있다.
  - 멱등 저장/해제 정책이라 프론트 낙관적 업데이트와 재시도 처리가 단순해진다.
- 부정적
  - 공개 조회 API라도 로그인 상태면 사용자별 후처리가 필요해 DTO 조합 로직이 조금 복잡해진다.
  - 관심 여부는 SSE 실시간 이벤트 자체에 포함되지 않아 저장/해제 상태 동기화는 화면 로컬 처리에 의존한다.
  - 관심 파티가 많아지면 사용자별 플래그 계산 비용을 더 줄이기 위한 전용 조회 전략이 필요할 수 있다.
- 중립적
  - 현재는 저장 여부만 다루며 메모, 태그, 폴더 같은 확장 속성은 후속 카드로 미룬다.
  - 홈 화면은 우선 탐색 중심으로 유지하고, 관심 토글 노출 범위는 목록/상세/내 관심 파티로 제한한다.
