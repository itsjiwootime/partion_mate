# ADR-YYYYMMDD-short-slug

- 날짜: YYYY-MM-DD
- 상태: Proposed | Accepted | Superseded
- 관련 작업: `E?-?`
- 관련 파일: `path/to/file`

## 배경
- 어떤 기술적 문제 또는 선택이 필요했는지 작성한다.
- 현재 구조의 한계, 성능 문제, 운영 문제, 복잡도 증가 요인을 구체적으로 적는다.

## 결정사항
- 어떤 결정을 내렸고 그 이유는 무엇인지 작성한다.
- 선택한 기술, 구조, 정책, 데이터 모델, 트랜잭션 경계, 테스트 전략 등을 명시한다.

## 대안
- 고려한 다른 선택지와 각각의 장단점을 작성한다.
- 최소 2개 이상의 대안을 비교하고, 왜 채택하지 않았는지 적는다.

## 결과
- 긍정적 트레이드오프를 작성한다.
- 부정적 트레이드오프를 작성한다.
- 중립적 트레이드오프 또는 후속 영향 사항을 작성한다.

## 예시 체크리스트
- 이 결정이 해결하려는 핵심 문제가 분명한가
- 선택 근거가 코드/성능/운영 관점에서 설명되는가
- 대안 비교가 형식적이지 않고 실제 선택지를 다루는가
- 결과 섹션에 긍정/부정/중립 영향이 모두 포함되는가
