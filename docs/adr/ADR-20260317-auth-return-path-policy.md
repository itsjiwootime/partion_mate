# ADR-20260317-auth-return-path-policy

- 날짜: 2026-03-17
- 상태: Accepted
- 관련 작업: `E9-3`
- 관련 파일: `frontend/src/pages/Login.jsx`, `frontend/src/pages/CreateParty.jsx`, `frontend/src/pages/JoinParty.jsx`, `frontend/src/pages/MyParties.jsx`, `frontend/src/pages/Notifications.jsx`, `frontend/src/pages/Profile.jsx`, `frontend/src/pages/Chat.jsx`

## 배경
- 보호 화면에서 비로그인 사용자를 로그인으로 보내더라도, 로그인 성공 후에는 항상 홈으로 이동해 직전 맥락이 끊기고 있었다.
- 참여, 채팅, 내 파티, 프로필처럼 의도가 분명한 화면에서 홈으로 튕기면 이탈 가능성이 높다.
- 로그인 복귀 경로를 각 화면이 제각각 처리하면 무한 리다이렉트나 로그인 화면 재진입 문제가 생길 수 있다.

## 결정사항
- 보호 화면은 로그인 이동 시 `state.from`에 현재 `pathname + search`를 저장한다.
- 로그인 화면은 `location.state?.from`을 읽어 복귀하되, `/login`과 `/signup`으로의 복귀는 허용하지 않고 홈으로 fallback한다.
- 사용자가 직접 로그인 화면에 들어온 경우에는 기존처럼 홈으로 이동한다.
- 자동 리다이렉트가 필요한 프로필/채팅 화면은 `replace: true`를 사용해 불필요한 브라우저 히스토리를 줄인다.

## 대안
- 로그인 후 항상 홈으로 이동 유지
  - 장점: 구현이 가장 단순하다.
  - 단점: 사용자의 의도가 끊기고 재탐색 비용이 크다.
- 쿼리스트링(`redirect=/...`)으로 복귀 경로 전달
  - 장점: 새로고침이나 외부 링크에도 비교적 명시적이다.
  - 단점: 보안과 인코딩 처리가 늘고, 현재 라우터 구조에서는 state 전달이 더 간단하다.
- 전역 인증 가드 컴포넌트에서만 일괄 처리
  - 장점: 코드 중복을 줄일 수 있다.
  - 단점: 현재 프로젝트는 화면별 보호 조건이 조금씩 달라 단일 가드 도입 범위가 더 커진다.

## 결과
- 긍정적
  - 로그인 이후 원래 하려던 행동으로 자연스럽게 복귀할 수 있다.
  - 화면별 접근 가드 동작이 일관되어 UX 단절이 줄어든다.
  - 복귀 경로 정책이 로그인 화면에 명확히 모여 예외 처리가 쉬워졌다.
- 부정적
  - 여러 화면이 로그인 이동 시 `state.from`을 전달해야 해 구현 포인트가 분산된다.
  - 새로고침으로 state가 사라지는 경우에는 홈 fallback이 유지된다.
- 중립적
  - 이후 서버 사이드 인증이나 외부 소셜 로그인 도입 시에는 복귀 경로 저장 방식을 다시 검토할 수 있다.
