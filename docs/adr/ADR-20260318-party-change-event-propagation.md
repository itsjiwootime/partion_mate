# ADR-20260318-party-change-event-propagation

- 날짜: 2026-03-18
- 상태: Accepted
- 관련 작업: `E13-3`
- 관련 파일: `src/main/java/com/project/partition_mate/service/PartyService.java`, `src/main/java/com/project/partition_mate/service/NotificationOutboxService.java`, `src/main/java/com/project/partition_mate/service/NotificationOutboxProcessor.java`, `frontend/src/pages/Chat.jsx`, `docs/specs/E13-party-change-events.md`

## 배경
- `E13-1`과 `E13-2`로 파티 수정/취소 API는 생겼지만, 수정 이벤트는 SSE, 앱 내 알림, 채팅 시스템 메시지로 함께 전파되지 않았다.
- 이 상태에서는 목록과 상세는 새로고침 전까지 예전 값을 볼 수 있고, 채팅방은 오픈채팅 링크나 제목 변경을 놓칠 수 있다.
- 특히 총 수량이 늘어날 때 대기열 승격이 수정 흐름과 분리돼 있으면, 호스트가 조건을 바꿔도 실제 참여 상태는 늦게 반영되는 문제가 생긴다.

## 결정사항
- 파티 수정 전용 전파 단위로 `PARTY_UPDATED` outbox 이벤트와 `PartyRealtimeTrigger.PARTY_UPDATED`를 추가한다.
- 수정 API는 변경 전 스냅샷과 수정 후 값을 비교해 `변경 항목: ...` 요약을 만들고, 이를 채팅 시스템 메시지와 앱 내 알림 본문에 공통으로 사용한다.
- 총 수량 증가가 감지되면 같은 수정 트랜잭션 안에서 기존 FIFO 대기열 승격 로직을 실행한다.
- `PARTY_UPDATED` 알림은 기존 참여자(호스트 제외)와 수정 후에도 남아 있는 대기자에게만 보낸다. 승격된 사용자는 `WAITING_PROMOTED` 알림으로 상태 변화를 받는다.
- 프론트 채팅 화면은 별도 상태 저장소를 두지 않고 기존 SSE 스트림을 재사용해, 관련 파티 이벤트가 오면 채팅방 목록과 상세 API를 다시 조회한다.

## 대안
- 수정 이벤트는 SSE만 보내고 앱 내 알림과 채팅 메시지는 생략한다.
  - 장점: 구현이 가장 단순하고 이벤트 타입이 적다.
  - 단점: 화면을 보고 있지 않은 사용자는 조건 변경을 놓치고, 채팅방 안에서도 수정 이력을 확인하기 어렵다.
- 수정 API 트랜잭션 안에서 직접 `UserNotification`을 저장한다.
  - 장점: 구조가 단순하고 별도 워커가 필요 없다.
  - 단점: 기존 outbox 재시도/중복 방지 정책과 분리돼 실패 복구와 일관성이 나빠진다.
- 채팅 화면은 웹소켓 시스템 메시지만 믿고 별도 SSE 재조회를 하지 않는다.
  - 장점: 채팅 화면 구현이 단순하다.
  - 단점: 오픈채팅 링크나 파티 제목처럼 채팅 메시지 payload에 없는 공개 정보는 즉시 갱신되지 않는다.

## 결과
- 긍정적
  - 파티 수정이 목록, 상세, 내 파티, 채팅, 앱 내 알림에 같은 의미로 전파돼 사용자 경험이 일관된다.
  - 수량 증가와 대기열 승격이 같은 흐름에서 처리돼 호스트 수정 직후 실제 참여 상태가 맞춰진다.
  - 알림 저장은 기존 outbox 워커를 재사용해 재시도와 중복 방지 정책을 유지한다.
- 부정적
  - 수정 1회에 `PARTY_UPDATED`, `WAITING_PROMOTED`, SSE, 채팅 시스템 메시지가 함께 발생할 수 있어 이벤트 수가 늘어난다.
  - 채팅 화면은 관련 이벤트마다 목록/상세를 다시 조회하므로 네트워크 호출이 증가한다.
- 중립적
  - 변경 항목 요약은 현재 사람이 읽기 쉬운 문자열 중심이라, 이후 외부 푸시나 관리자 도구에서 구조화된 필드 diff가 필요하면 payload를 확장할 수 있다.
  - 채팅 화면은 여전히 사용자 전용 상태를 API 재조회로 맞추므로, 향후 채널 수가 더 늘어나면 전역 상태 저장소를 다시 검토할 수 있다.
