# ADR-20260316-party-realtime-frontend-sync

- 날짜: 2026-03-16
- 상태: Accepted
- 관련 작업: `E4-3`
- 관련 파일: `frontend/src/utils/party.js`, `frontend/src/utils/partyRealtime.js`, `frontend/src/pages/PartyList.jsx`, `frontend/src/pages/PartyDetail.jsx`, `frontend/src/pages/MyParties.jsx`

## 배경
- 같은 파티 상태가 목록, 상세, 내 파티 화면에 각각 다른 형태로 저장돼 있으면 실시간 이벤트를 그대로 적용하기 어렵다.
- 파티 목록과 상세는 공개 스냅샷을 바로 병합할 수 있지만, 내 파티 화면은 사용자별 역할과 대기 상태가 섞여 있어 단순 병합만으로는 일관성이 깨질 수 있다.
- 실시간 반영은 빠르게 보이되, 기존 API 응답 구조와 충돌하지 않는 전략이 필요했다.

## 결정사항
- 프론트는 `subscribeToPartyStream` 유틸을 통해 SSE 구독과 재연결 상태 관리를 공통 처리한다.
- 파티 목록과 상세는 `mergeRealtimeParty`, `applyPartyListRealtimeUpdate`로 이벤트 payload를 기존 정규화 구조에 병합한다.
- 내 파티 화면은 공개 이벤트를 직접 병합하지 않고, 현재 목록에 포함된 파티 ID와 일치할 때 `/api/users/me/parties`를 다시 조회한다.
- 화면에는 `live/reconnecting` 상태 문구를 추가해 실시간 연결 상태를 사용자에게 노출한다.

## 대안
- 모든 화면에서 이벤트 payload만 직접 병합
  - 장점: API 재조회가 줄어든다.
  - 단점: 내 파티의 사용자별 역할, 대기열, 요청 수량 같은 필드는 공개 이벤트만으로 복원하기 어렵다.
- 모든 이벤트에서 모든 화면이 API 재조회
  - 장점: 상태 정합성이 높다.
  - 단점: 실시간 이벤트가 많은 경우 API 부하가 커지고 반응성이 떨어진다.
- 전역 상태 저장소 도입
  - 장점: 여러 화면 간 상태 공유가 쉬워진다.
  - 단점: 현재 프로젝트 규모 대비 상태 관리 복잡도가 과하다.

## 결과
- 긍정적
  - 공개 화면은 즉시 반응하고, 사용자 전용 화면은 정확한 API 재조회로 정합성을 확보했다.
  - 기존 `normalizePartyDetail` 구조를 유지하면서도 실시간 이벤트를 재사용할 수 있게 됐다.
  - 전역 상태 저장소 없이도 에픽 4 범위를 충족했다.
- 부정적
  - 내 파티 화면은 이벤트마다 재조회가 발생하므로 공개 화면보다 네트워크 비용이 크다.
  - 화면별 반영 전략이 달라 구현 포인트가 조금 늘었다.
- 중립적
  - 이후 읽음 처리나 채팅처럼 사용자 전용 실시간 데이터가 늘어나면 전역 상태 또는 별도 채널 도입을 다시 검토할 수 있다.
