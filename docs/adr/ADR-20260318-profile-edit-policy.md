# ADR-20260318-profile-edit-policy

- 날짜: 2026-03-18
- 상태: Accepted
- 관련 작업: `E18-1`
- 관련 파일: `src/main/java/com/project/partition_mate/controller/UserController.java`, `src/main/java/com/project/partition_mate/service/UserService.java`, `frontend/src/pages/Profile.jsx`

## 배경
- 현재 사용자 프로필은 조회만 가능하고, 닉네임과 주소를 바꾸려면 재가입 수준의 우회가 필요했다.
- 홈 화면은 저장된 좌표를 기준으로 주변 지점을 불러오기 때문에, 주소 문자열만 바뀌고 좌표가 남아 있으면 사용자에게 더 잘못된 위치 기준을 보여줄 수 있었다.
- 프로필 저장 후 전역 사용자명 상태를 다시 맞추지 않으면 헤더와 채팅 화면에 이전 닉네임이 남는 문제가 있었다.

## 결정사항
- `PATCH /api/users/me`를 추가해 닉네임, 주소, 선택적 좌표 쌍을 수정할 수 있게 한다.
- 닉네임은 다른 사용자가 이미 사용 중이면 `409 CONFLICT`로 거절한다.
- 주소가 바뀌었는데 좌표를 함께 보내지 않으면 기존 저장 좌표를 초기화한다.
- 프론트는 프로필 저장 성공 후 `AuthContext.refreshProfile()`을 호출해 전역 사용자명과 이메일 캐시를 다시 맞춘다.

## 대안
- 주소 수정 자체를 `E18-2`까지 미루는 방식
  - 장점: 주소와 좌표를 한 번에 다루는 UX만 구현하면 된다.
  - 단점: 사용자가 닉네임과 기본 주소도 수정하지 못한 상태가 길어져 설정 화면 가치가 낮다.
- 주소가 바뀌어도 기존 좌표를 유지하는 방식
  - 장점: 홈 화면의 저장 좌표 기반 탐색이 계속 동작한다.
  - 단점: 주소 문자열과 실제 탐색 기준 좌표가 어긋나 더 잘못된 결과를 줄 수 있다.
- 저장 후 프로필 화면 로컬 상태만 갱신하는 방식
  - 장점: 구현이 단순하다.
  - 단점: 헤더와 채팅 등 다른 화면에 이전 닉네임이 남아 전역 일관성이 깨진다.

## 결과
- 긍정적 트레이드오프: 사용자가 닉네임과 주소를 직접 수정할 수 있고, 저장 직후 전역 사용자명이 일관되게 갱신된다.
- 부정적 트레이드오프: 주소만 수정하면 저장 좌표가 초기화되어 홈 화면이 기본 위치나 브라우저 위치 fallback을 사용할 수 있다.
- 중립적 트레이드오프 또는 후속 영향 사항: 정확한 좌표 재설정 UX는 `E18-2`에서 이어서 보강해야 하며, 이번 카드는 텍스트 프로필 수정과 상태 일관성 확보에 집중한다.
