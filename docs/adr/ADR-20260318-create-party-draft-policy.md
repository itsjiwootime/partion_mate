# ADR-20260318-create-party-draft-policy

- 날짜: 2026-03-18
- 상태: Accepted
- 관련 작업: `E17-3`
- 관련 파일: `frontend/src/pages/CreateParty.jsx`, `frontend/src/utils/createPartyDraft.js`, `frontend/src/utils/createPartyDraft.test.js`

## 배경
- 파티 생성 화면은 3단계 구조가 되었지만, 입력 도중 새로고침하거나 다른 화면으로 이동하면 사용자가 작성 중이던 내용을 다시 입력해야 했다.
- 특히 지점, 상품명, 가격, 수량, 마감 시간처럼 서로 다른 단계에 흩어진 입력이 많아 한 번 이탈하면 복구 비용이 컸다.
- 초안 복구가 없으면 단계형 폼의 장점은 유지되더라도 이탈 복원이라는 핵심 사용자 경험이 비어 있었다.

## 결정사항
- 생성 화면 초안은 브라우저 `localStorage`에 저장하되, 키를 로그인 사용자 기준으로 분리한다.
- 저장 payload에는 `version`, `savedAt`, `currentStep`, `storeId`, `form`을 함께 보관해 스키마 버전, 마지막 진행 단계, 작성 중이던 지점을 같이 복구한다.
- 저장은 사용자가 복구 배너에 응답한 뒤부터 자동 저장하고, 초기값과 동일한 상태가 되면 초안을 삭제한다.
- 초안은 7일 TTL을 두고, 버전이 다르거나 JSON 구조가 맞지 않으면 자동 삭제한다.
- 복구 UX는 자동 적용이 아니라 배너 기반의 명시적 선택으로 처리해, 사용자가 현재 지점 기준 새 입력을 시작할 여지를 남긴다.
- 같은 사용자라도 URL의 `storeId`와 저장된 초안의 `storeId`가 다르면 자동 복구 배너를 노출하지 않는다.
- 예전 전역 키(`pm_create_party_draft`)에 남은 초안은 더 이상 읽지 않고 정리한다.

## 대안
- 서버에 초안을 저장하는 방식
  - 장점: 기기 간 동기화와 계정 단위 복구가 가능하다.
  - 단점: 이번 범위에서는 백엔드 저장 모델과 인증 경로를 추가해야 해 구현 비용이 크다.
- 브라우저 전체에서 전역 키 하나만 공유하는 방식
  - 장점: 구현이 가장 단순하다.
  - 단점: 로그아웃 후 다른 사용자가 같은 브라우저를 써도 이전 초안을 보게 되어 계정 경계가 무너진다.
- 진입 즉시 초안을 자동 복구하는 방식
  - 장점: 사용자가 별도 클릭 없이 바로 이어서 작성할 수 있다.
  - 단점: 현재 쿼리 파라미터나 새로 선택한 지점 기준으로 새 파티를 만들려는 사용자의 의도와 충돌할 수 있다.
- `sessionStorage`를 사용하는 방식
  - 장점: 탭 종료 시 자동 정리되어 저장소가 덜 남는다.
  - 단점: 탭을 닫은 뒤 다시 돌아왔을 때 복구가 되지 않아 이 카드의 목적과 맞지 않는다.

## 결과
- 긍정적 트레이드오프: 생성 화면 이탈 후에도 직전 단계와 입력값을 복구할 수 있고, 다른 로그인 사용자가 같은 브라우저를 써도 초안이 섞이지 않는다.
- 부정적 트레이드오프: `localStorage`에 사용자 입력이 일정 기간 남기 때문에 브라우저 공유 환경에서는 민감하지 않은 입력만 저장한다는 전제가 여전히 필요하다.
- 중립적 트레이드오프 또는 후속 영향 사항: 스키마가 바뀌면 `version`을 올리고 복구 유틸과 관련 테스트를 함께 갱신해야 하며, 서버 초안 동기화가 필요해지면 현재 로컬 스코프 정책을 다시 검토해야 한다.
