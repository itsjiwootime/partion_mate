# ADR-20260318-internal-chat-only-policy

- 날짜: 2026-03-18
- 상태: Accepted
- 관련 작업: `E17-1`
- 관련 파일: `frontend/src/pages/CreateParty.jsx`, `frontend/src/pages/Chat.jsx`, `frontend/src/pages/PartyDetail.jsx`, `src/main/java/com/project/partition_mate/dto/CreatePartyRequest.java`, `src/main/java/com/project/partition_mate/dto/UpdatePartyRequest.java`

## 배경
- 에픽 8에서 파티 생성 시 전용 인앱 채팅방이 자동 생성되는 구조가 이미 도입되었다.
- 그럼에도 생성 폼과 일부 화면에는 외부 오픈채팅 링크 fallback이 남아 있어, 사용자가 내부 채팅 대신 외부 채널로 이탈할 수 있었다.
- 신고, 차단, 공지, unread, 시스템 메시지 같은 안전/운영 기능을 내부 채팅으로 일관되게 묶으려면 외부 fallback 의존을 줄여야 했다.

## 결정사항
- 신규 파티 생성과 수정 요청에서는 외부 오픈채팅 링크를 더 이상 받지 않는다.
- 파티 상세와 채팅 화면에서 외부 오픈채팅 fallback UI를 제거하고, 인앱 채팅을 유일한 기본 대화 채널로 둔다.
- 정산 또는 픽업 이후의 파티 수정 제한도 `guideNote` 단일 수정 허용으로 단순화한다.
- 기존 도메인 컬럼은 즉시 물리 삭제하지 않고, 사용자 흐름과 API 계약에서 먼저 제거하는 방향으로 정리한다.

## 대안
- 외부 오픈채팅을 계속 선택 옵션으로 유지한다.
  - 장점: 기존 운영 습관을 그대로 가져갈 수 있다.
  - 단점: 신고, 차단, 공지, 읽지 않음, 시스템 메시지 같은 내부 기능이 우회되고 일관성이 깨진다.
- 신규 파티만 내부 채팅 전용으로 두고 기존 파티는 fallback을 유지한다.
  - 장점: 기존 데이터와의 급격한 단절이 줄어든다.
  - 단점: 화면/정책이 파티 생성 시점에 따라 달라져 사용자가 이해하기 어렵다.
- 도메인 컬럼과 DTO를 한 번에 모두 삭제한다.
  - 장점: 기술 부채가 즉시 줄어든다.
  - 단점: 관련 쿼리, 응답, 기존 데이터 호환성까지 한 번에 깨질 수 있어 범위가 커진다.

## 결과
- 긍정적
  - 대화, 공지, unread, 차단/신고 흐름이 인앱 채팅 기준으로 더 일관된다.
  - 생성 폼과 파티 수정 정책이 단순해져 사용자가 선택해야 할 항목이 줄어든다.
  - 이후 안전 기능과 운영 도구를 내부 채팅 기준으로 확장하기 쉬워진다.
- 부정적
  - 기존 외부 오픈채팅 운영 습관이 있던 사용자는 전환 비용을 느낄 수 있다.
  - 도메인 컬럼이 당장 삭제되지는 않으므로 기술적으로는 과도기 상태가 남는다.
  - 기존 파티 데이터에 남아 있는 오픈채팅 링크는 UI에서 보이지 않아도 DB에는 잔존할 수 있다.
- 중립적
  - 이 결정은 `ADR-20260317-chat-ui-and-openchat-fallback`를 대체한다.
  - 추후 DB 마이그레이션이나 응답 DTO 정리는 별도 카드에서 이어서 처리할 수 있다.
