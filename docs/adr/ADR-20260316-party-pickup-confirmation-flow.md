# ADR-20260316-party-pickup-confirmation-flow

- 날짜: 2026-03-16
- 상태: Accepted
- 관련 작업: `E6-3`
- 관련 파일: `src/main/java/com/project/partition_mate/domain/Party.java`, `src/main/java/com/project/partition_mate/domain/PartyMember.java`, `src/main/java/com/project/partition_mate/service/PartyService.java`, `frontend/src/pages/PartyDetail.jsx`

## 배경
- 공동구매 파티가 모집 완료되더라도 언제 어디서 물건을 주고받을지 정해지지 않으면 실제 거래는 성사되지 않는다.
- 픽업 정보는 호스트가 정하지만, 참여자가 내용을 확인했는지까지 추적해야 이후 거래 완료/노쇼 판정을 설득력 있게 할 수 있다.
- 오픈채팅만으로 픽업 정보를 전달하면 서비스 내부 상태와 운영 흐름이 끊기므로, 파티 도메인 안에서 일정 정보를 관리할 필요가 있었다.

## 결정사항
- 파티 자체에 `pickupPlace`, `pickupTime`을 저장하고, 호스트만 `PUT /party/{id}/pickup`으로 값을 확정할 수 있게 한다.
- 참여자별 확인 여부는 `PartyMember.pickupAcknowledgedAt`으로 저장하고, 참여자는 `POST /party/{id}/pickup/acknowledge`로 본인 확인 상태를 남긴다.
- 상세 화면에는 호스트용 픽업 일정 입력 폼과 참여자용 확인 버튼을 같은 화면에 배치해 별도 운영 페이지 없이 흐름을 끝낸다.
- 픽업 일정은 실시간 이벤트와 상세 재조회로 동기화하고, `MyParties`에서도 장소/시간을 요약 노출한다.

## 대안
- 픽업 정보를 별도 엔티티로 관리
  - 장점: 장소 변경 이력이나 다중 일정 지원에 유리하다.
  - 단점: 현재는 단일 픽업 일정만 필요해 구조가 과하다.
- 픽업 확인 여부를 파티 단위 boolean 하나로만 저장
  - 장점: 구현이 단순하다.
  - 단점: 어떤 참여자가 확인했는지 추적할 수 없어 노쇼 처리와 연결하기 어렵다.
- 서비스 안에서는 픽업 정보를 저장하지 않고 채팅/오픈채팅에만 의존
  - 장점: 개발 비용이 작다.
  - 단점: 픽업 운영 상태가 시스템 밖으로 빠져나가 후속 자동화와 UI 표시가 불가능해진다.

## 결과
- 긍정적
  - 픽업 일정과 참여자 확인 여부가 서비스 내부 상태로 관리돼 거래 진행도가 명확해졌다.
  - 호스트와 참여자가 한 화면에서 필요한 액션을 처리할 수 있어 UX가 단순해졌다.
  - 이후 채팅/알림과도 쉽게 연결할 수 있는 기준 데이터가 생겼다.
- 부정적
  - 상세 화면이 운영 기능까지 맡으면서 UI 복잡도가 올라갔다.
  - 현재는 단일 픽업 일정만 지원하므로, 장소 변경 이력은 별도로 남지 않는다.
- 중립적
  - 이후 에픽 8 채팅에서 픽업 공지 고정을 구현할 때 이 필드를 공지 생성의 소스 데이터로 재사용할 수 있다.
