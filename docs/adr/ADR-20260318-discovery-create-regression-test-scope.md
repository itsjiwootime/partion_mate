# ADR-20260318-discovery-create-regression-test-scope

- 날짜: 2026-03-18
- 상태: Accepted
- 관련 작업: `E20-1`
- 관련 파일: `frontend/src/pages/DiscoveryCreateFlow.test.jsx`, `frontend/src/pages/Home.quickDiscovery.test.jsx`, `frontend/src/pages/PartyList.filters.test.jsx`, `frontend/src/pages/CreateParty.test.jsx`

## 배경
- `E16`과 `E17`을 거치며 홈 탐색, 지점별 파티 목록, 생성 폼 단계화가 각각 강화됐지만, 화면 간 실제 이동 경로를 묶어 보는 자동 테스트는 없었다.
- 그 결과 홈에서 선택한 지점이 생성 화면까지 유지되는지, `PartyList`의 `파티 만들기` 진입이 올바른 쿼리를 넘기는지 같은 연결부 회귀를 기존 테스트만으로는 보장하기 어려웠다.
- 또한 `PartyList`는 토스트 컨텍스트에 의존하는데 일부 테스트가 그 의존성을 반영하지 않아, 런타임과 테스트 환경이 어긋날 여지가 있었다.

## 결정사항
- `Vitest + React Testing Library` 기반은 그대로 유지하되, `Home -> PartyList -> CreateParty`를 실제 `MemoryRouter` 라우트 위에 올리는 라우터 통합 성격 테스트를 추가한다.
- 기존 페이지별 테스트는 유지하고, 검색/필터/생성 단계/초안 복구 같은 화면 내부 동작 검증은 각 화면 테스트 파일에 둔다.
- `PartyList` 테스트는 `ToastContext` mock을 공통 의존성으로 포함해 컴포넌트가 실제 런타임과 같은 조건에서 렌더링되게 맞춘다.
- 이번 카드에서는 브라우저 E2E 러너를 도입하지 않고, 빠른 회귀 피드백이 가능한 라우터 레벨 테스트까지만 확장한다.

## 대안
- 기존 `Home`, `PartyList`, `CreateParty` 테스트 파일에 개별 assertion만 더 추가한다.
  - 장점: 새 테스트 파일 없이 현재 구조를 유지할 수 있다.
  - 단점: 실제 페이지 간 handoff를 한 번에 보지 못해, 쿼리 전달과 선택 지점 유지 같은 연결 회귀가 분산된 assertion에 묻힌다.
- Playwright나 Cypress 같은 E2E 러너를 도입한다.
  - 장점: 브라우저 권한, 네트워크, 라우팅을 더 실제에 가깝게 검증할 수 있다.
  - 단점: `E20-1` 범위를 넘어 인프라와 유지비가 커지고, 현재 필요한 빠른 회귀 검증보다 과한 선택이다.
- 생성 화면 테스트만 강화하고 홈/지점 목록 연결은 수동 확인에 맡긴다.
  - 장점: 가장 적은 비용으로 생성 폼 자체는 더 촘촘하게 볼 수 있다.
  - 단점: 탐색에서 생성으로 이어지는 핵심 진입 경로가 자동 검증 범위 밖에 남는다.

## 결과
- 긍정적
  - 홈에서 선택한 지점이 지점 파티 목록과 생성 화면까지 정확히 이어지는 핵심 경로를 자동으로 검증할 수 있다.
  - 기존 검색/필터/생성 테스트와 새 라우터 흐름 테스트가 분리되어, 실패 시 어느 계층에서 깨졌는지 더 빠르게 파악할 수 있다.
  - `PartyList` 테스트가 실제 의존성과 맞춰져 테스트 환경과 런타임의 차이로 인한 거짓 실패 가능성을 줄인다.
- 부정적
  - 화면 3개를 한 번에 올리는 테스트라 mock 준비가 더 길고, API 의존성이 늘어난다.
  - 브라우저 권한, 실제 주소 검색 팝업, 네트워크 지연 같은 완전한 사용자 환경까지 검증하지는 못한다.
  - 라우트 구조나 공통 컨텍스트가 바뀌면 테스트 보수 비용이 증가한다.
- 중립적
  - 이번 결정은 프론트 테스트 러너 선택을 바꾸지 않고, 기존 E12 기반 위에 회귀 범위만 확장하는 성격이다.
  - 이후 `E20-2`, `E20-3`도 같은 방식으로 페이지 간 핵심 handoff를 우선 자동화하고, 브라우저 E2E는 필요성이 커질 때 별도 도입할 수 있다.
