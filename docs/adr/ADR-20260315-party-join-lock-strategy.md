# ADR-20260315 Party Join Lock Strategy

## 배경
파티 참여 요청은 동시에 들어올 수 있고, 같은 시점에 남은 수량을 읽고 저장하면 총수량을 초과한 상태로 모집이 완료될 수 있다. 또한 중복 참여가 API 계층 검증을 통과하더라도 DB flush 시점 충돌로 뒤늦게 드러날 수 있어, 서비스 계층에서 일관된 예외 응답을 보장할 필요가 있었다.

## 결정사항
`joinParty`는 `PartyRepository.findByIdForUpdate`의 `PESSIMISTIC_WRITE` 락으로 파티 행을 잠그고, 트랜잭션 안에서 남은 수량 검증과 상태 전이를 수행한다. 참여자는 `party.acceptMember`로 도메인 규칙을 먼저 적용한 뒤 `partyMemberRepository.saveAndFlush`로 즉시 DB 제약을 확인하고, 유니크 제약 충돌은 `BusinessException.alreadyJoined()`로 변환해 공통 예외 응답을 유지한다.

## 대안
대안 1은 낙관적 락을 적용하는 방식이었다. 구현은 덜 무겁지만, 현재 참여 API는 같은 파티에 순간적으로 요청이 몰릴 가능성이 높고 재시도 정책까지 같이 설계하지 않으면 충돌 처리 복잡도가 커진다.

대안 2는 애플리케이션 메모리에서만 현재 인원 수를 계산하고 마지막에 저장하는 방식이었다. 코드 변경은 적지만, 다중 요청에서 정원 초과를 막을 수 없고 이미 확인한 동시성 문제를 해결하지 못한다.

대안 3은 Redis 분산락을 바로 도입하는 방식이었다. 다중 인스턴스 확장에는 유리하지만, 현재는 단일 DB 기반 구조에서 해결 가능한 문제를 외부 인프라로 확장하는 셈이라 운영 복잡도가 앞선다.

## 결과
긍정적으로는 정원 초과를 DB 트랜잭션 단위에서 막을 수 있고, 중복 참여도 서비스 계층에서 일관된 `409 CONFLICT` 응답으로 변환할 수 있다. 기존 통합 테스트와 새 예외 응답 테스트로 동시성 및 실패 응답을 함께 검증할 수 있다.

부정적으로는 비관적 락이므로 동일 파티에 대한 경쟁이 심할수록 대기 시간이 길어질 수 있다. 향후 트래픽이 커지면 Redis 분산락이나 대기열 기반 비동기화로 재검토할 가능성이 있다.

중립적으로는 현재 단계에서 락의 범위를 파티 단건으로 제한해 구현 단순성을 유지했다. 다중 파티 교차 참여나 분산 환경 확장성은 이후 벤치마크 결과를 보고 판단해야 한다.
