# ADR-20260318-login-failure-policy-and-validation

- 날짜: 2026-03-18
- 상태: Accepted
- 관련 작업: `E11-5`
- 관련 파일: `src/main/java/com/project/partition_mate/service/AuthService.java`, `src/main/java/com/project/partition_mate/dto/LoginRequest.java`, `frontend/src/pages/Login.jsx`

## 배경
- 로그인 실패 시 존재하지 않는 이메일과 비밀번호 오류가 서로 다른 응답으로 내려가면 계정 존재 여부를 추정할 수 있다.
- 현재 로그인 검증 문구는 실제 제약과 맞지 않아 사용자가 어떤 입력이 허용되는지 혼란을 겪을 수 있다.
- 프론트 로그인 화면은 서버 에러를 그대로 보여주지만, 입력 정책에 대한 사전 안내가 부족해 실패 후에만 규칙을 알게 된다.

## 결정사항
- 존재하지 않는 이메일과 비밀번호 불일치를 모두 `401 LOGIN_FAILED`와 `이메일 또는 비밀번호를 확인해주세요.` 문구로 통일한다.
- 로그인 실패 분기에서도 `passwordEncoder.matches`를 더미 해시와 함께 호출해, 이메일 존재 여부에 따른 분기 차이를 줄인다.
- 로그인 비밀번호 검증 문구를 실제 정책과 맞춰 `6자 이상 20자 이하`로 수정한다.
- 프론트 로그인 화면에 동일한 비밀번호 길이 안내 문구를 추가해, 제출 전에도 사용자가 정책을 확인할 수 있게 한다.

## 대안
- 이메일 미존재는 404, 비밀번호 오류는 401로 구분 유지한다.
  - 장점: 디버깅과 사용자 문의 대응이 단순하다.
  - 단점: 계정 존재 여부가 그대로 노출돼 보안적으로 불리하다.
- 모든 로그인 실패를 같은 메시지로만 묶고 더미 해시는 사용하지 않는다.
  - 장점: 구현이 가장 단순하고 응답 정책은 통일된다.
  - 단점: 코드 경로 차이가 남아 있어 타이밍 기반 추정 여지를 완전히 줄이지 못한다.
- 프론트에서만 메시지를 감추고 백엔드 응답은 구분된 상태로 둔다.
  - 장점: 백엔드 로직을 덜 바꾸고 화면 문구만 통제할 수 있다.
  - 단점: API 자체는 여전히 계정 존재 여부를 노출하므로 근본 해결이 아니다.

## 결과
- 긍정적 트레이드오프: 로그인 실패 정책이 단순해지고, API 응답만으로 계정 존재 여부를 추정하기 어려워진다.
- 부정적 트레이드오프: 내부적으로는 사용자를 더 빨리 진단할 수 있는 정보가 줄어들어, 운영 디버깅 시 별도 로그나 추적이 더 중요해진다.
- 중립적 트레이드오프: 더미 해시 비교와 안내 문구 추가로 로그인 코드와 UI가 조금 길어졌지만, 현재 프로젝트 규모에서는 유지 비용이 크지 않다.
