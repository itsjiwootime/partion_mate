# ADR-20260318-party-discovery-sort-policy

- 날짜: 2026-03-18
- 상태: Accepted
- 관련 작업: `E16-2`
- 관련 파일: `frontend/src/utils/partyDiscovery.js`, `frontend/src/pages/PartyList.jsx`, `frontend/src/pages/Home.jsx`, `frontend/src/components/PartyCard.jsx`

## 배경
- `E16-1`로 검색과 필터는 가능해졌지만, 결과가 여러 개일 때 어떤 파티를 먼저 보여줄지에 대한 기준이 없었다.
- 현재 파티 요약 응답에는 `deadline`, `currentQuantity`, `totalQuantity`, `partyId`는 있지만 `createdAt`이나 서버 정렬 API는 없다.
- 따라서 이번 단계에서는 기존 응답 범위 안에서 동작하는 프론트 정렬 정책과 발견 섹션 기준이 필요했다.

## 결정사항
- 파티 목록 정렬은 URL 쿼리 `sort`로 관리하고, `recommended`, `deadline`, `popular`, `newest` 네 가지 모드를 지원한다.
- 정렬 계산은 `partyDiscovery` 유틸에서 프론트 공통 로직으로 처리하고 홈과 파티 목록이 같은 기준을 공유한다.
- 기본 정렬은 `recommended`로 두고, 상태 우선순위 후 `마감 임박 -> 인기 -> 최신` 순으로 tie-break 한다.
- `newest`는 현재 응답에 생성 시각이 없으므로 `partyId` 내림차순으로 대체한다.
- 발견 섹션은 `마감 임박`, `인기 파티`, `신규 파티` 세 카드로 고정하고, 현재 조건에서 대표 파티 1개를 보여준 뒤 해당 정렬로 이동하게 한다.
- 현재 정렬 결과 상위 3개 카드에는 기준에 맞는 강조 badge를 붙인다.

## 대안
- 백엔드 정렬 API와 전용 discovery endpoint를 먼저 추가한다.
  - 장점: 데이터가 많아져도 서버에서 일관된 정렬과 페이징을 함께 처리할 수 있다.
  - 단점: 이번 카드 목표인 탐색 UX 보강보다 범위가 커지고, 현재 자유입력 상품 구조에서는 정렬 가치가 제한적이다.
- 기본 정렬 없이 사용자가 매번 직접 선택하게 둔다.
  - 장점: 정책 논쟁이 줄고 구현이 단순하다.
  - 단점: 목록 첫 인상이 매번 달라지고, 홈 발견 카드와 파티 목록의 연결 기준이 약해진다.
- `newest`를 임의의 프론트 수신 순서에 의존한다.
  - 장점: 구현이 가장 단순하다.
  - 단점: 새로고침이나 SSE 재동기화 후 순서가 흔들려 최신 기준으로 설명할 수 없다.

## 결과
- 긍정적
  - 서버 변경 없이 홈과 파티 목록 모두에서 일관된 정렬과 발견 진입점을 제공한다.
  - `sort` 쿼리 기반이라 링크 공유, 새로고침, 홈 진입 이후에도 같은 정렬 상태를 복원할 수 있다.
  - 상위 카드 강조 규칙이 생겨 현재 어떤 기준으로 추천되는지 사용자에게 더 명확하게 보인다.
- 부정적
  - 프론트 계산 정렬이라 서버 기준 정렬과 다를 수 있고, 데이터 규모가 커지면 한계가 있다.
  - `newest`를 `partyId`로 추정하는 것은 근사치일 뿐 실제 생성 순서를 보장하지 않는다.
  - 홈에서 발견 섹션을 위해 `GET /party/all`을 추가로 호출하므로 초기 로딩 비용이 조금 늘어난다.
- 중립적
  - 홈의 `내 주변 지점`은 계속 거리순으로 유지해, 파티 정렬과 지점 정렬을 분리된 축으로 다룬다.
  - 추후 `createdAt`이나 서버 정렬 API가 추가되면 같은 `sort` 모델을 유지한 채 내부 계산만 교체할 수 있다.
