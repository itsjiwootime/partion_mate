# ADR-20260318-create-party-step-structure

- 날짜: 2026-03-18
- 상태: Accepted
- 관련 작업: `E17-1`
- 관련 파일: `frontend/src/pages/CreateParty.jsx`

## 배경
- 파티 생성 화면은 지점, 제품명, 가격, 수량, 소분 조건, 마감, 안내 문구를 한 화면에서 모두 입력하게 되어 있었다.
- 이 구조는 모바일에서 스크롤이 길고, 처음 사용하는 사용자가 어떤 순서로 입력해야 하는지 이해하기 어렵다.
- 동시에 현재 서비스는 파티 생성 시 전용 인앱 채팅방이 자동 생성되므로, 외부 오픈채팅 입력은 생성 폼 복잡도만 높이고 있었다.

## 결정사항
- 생성 폼을 `기본 정보 -> 소분 조건 -> 운영 정보` 3단계 구조로 나눈다.
- 상단 단계 카드와 현재 단계 진행 표시를 제공하고, 미래 단계 직접 이동은 막는다.
- 단계 이동은 현재 단계 검증을 통과한 뒤에만 허용한다.
- 마지막 단계에 최종 확인 요약을 함께 보여주고, 여기서 바로 생성 요청을 보낸다.
- 파티 생성 폼에서는 외부 오픈채팅 입력을 제거하고, 생성 후 인앱 채팅방 자동 생성 문구를 명시한다.

## 대안
- 기존 단일 폼을 유지하고 helper text만 추가한다.
  - 장점: 구현이 가장 단순하고 숙련 사용자 입력 속도가 빠르다.
  - 단점: 모바일 스크롤과 입력 피로가 그대로 남고, 필수값 누락을 단계적으로 막기 어렵다.
- 4단계 이상 세분화된 wizard로 나눈다.
  - 장점: 각 단계 집중도가 더 높아진다.
  - 단점: 현재 입력 항목 수 대비 단계가 과도하게 많아져 숙련 사용자에게 답답한 흐름이 된다.
- 단계형 없이 accordions 형태로 접고 펼친다.
  - 장점: 한 화면 구조를 유지하면서 구역만 분리할 수 있다.
  - 단점: 실제 입력 순서와 검증 시점이 약해지고, 모바일에서 열린 섹션이 많아지면 다시 길어진다.

## 결과
- 긍정적
  - 사용자가 입력해야 할 정보를 맥락별로 나눠 이해하기 쉬워진다.
  - 필수 입력 누락을 단계별로 막아 한 번에 많은 오류를 맞는 경험을 줄일 수 있다.
  - 오픈채팅 입력 제거로 생성 폼이 현재 서비스 구조와 더 일관되게 맞춰진다.
- 부정적
  - 숙련 사용자는 한 번에 모두 입력하는 방식보다 단계 이동이 느리다고 느낄 수 있다.
  - 단계 상태와 검증 메시지 관리가 추가되어 컴포넌트 복잡도가 증가한다.
  - 초안 저장이 없는 상태라 중간 이탈 시 단계 구조만으로는 복구를 해결하지 못한다.
- 중립적
  - 예상 금액, 경고 문구, 조건 미리보기는 기존 계산을 유지하고, 더 적극적인 안내는 `E17-2`에서 확장한다.
  - 단계 수는 현재 3개로 고정했지만, 추후 입력 항목이 늘어나면 같은 패턴을 유지한 채 재조정할 수 있다.
