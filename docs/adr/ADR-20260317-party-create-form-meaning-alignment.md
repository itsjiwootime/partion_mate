# ADR-20260317-party-create-form-meaning-alignment

- 날짜: 2026-03-17
- 상태: Accepted
- 관련 작업: `E9-2`
- 관련 파일: `frontend/src/pages/CreateParty.jsx`

## 배경
- 파티 생성 폼의 날짜/시간 입력은 사용자에게 모임 시각처럼 보였지만 실제로는 `deadline`으로 저장되고 있었다.
- 카테고리 필드는 UI에 있었지만 생성 payload에 포함되지 않아 사용자가 입력해도 저장되지 않았다.
- 호스트 요청 수량은 0도 유효한 값이지만, 프론트에서 falsy 처리로 1로 강제될 여지가 있었다.

## 결정사항
- 생성 폼 라벨을 `모임 날짜/시간`에서 `모집 마감 날짜/시간`으로 명확히 바꾸고, 픽업 일정은 파티 생성 후 별도 확정된다는 도움말을 추가한다.
- 저장되지 않는 카테고리 필드는 이번 단계에서 제거하고, 생성 화면에 남는 입력값은 모두 실제 payload와 1:1로 연결한다.
- 호스트 요청 수량은 0도 그대로 전송되도록 숫자 변환만 수행하고, 별도 fallback 기본값을 두지 않는다.
- 제품명/제목, 외부 오픈채팅 링크도 실제 사용 위치를 설명하는 보조 문구를 추가해 입력 의도를 맞춘다.

## 대안
- 현재 라벨을 유지하고 백엔드 필드명만 문서로 설명
  - 장점: 구현 변경이 적다.
  - 단점: 화면 자체가 오해를 유발해 문서로 보완해도 UX 문제가 남는다.
- 카테고리를 백엔드까지 즉시 연동
  - 장점: 입력 필드를 유지할 수 있다.
  - 단점: 현재 도메인과 활용처가 정리되지 않은 상태에서 불필요한 모델이 늘어난다.
- 픽업 시간 필드를 생성 폼에 같이 추가
  - 장점: 한 화면에서 더 많은 정보를 입력할 수 있다.
  - 단점: 현재 도메인 정책상 픽업은 모집 이후 호스트가 확정하는 흐름과 충돌한다.

## 결과
- 긍정적
  - 사용자가 입력하는 값과 실제 저장 의미가 일치하게 됐다.
  - 저장되지 않는 필드를 제거해 폼 신뢰도를 높였다.
  - 이후 픽업/정산 에픽과 생성 폼의 역할 경계가 분명해졌다.
- 부정적
  - 카테고리 분류 기능을 원하던 사용자에게는 입력 항목이 줄어든 것처럼 보일 수 있다.
  - 생성 화면의 설명 문구가 늘어 레이아웃이 조금 길어졌다.
- 중립적
  - 카테고리가 실제 추천, 검색, 통계 기능으로 필요해지면 별도 에픽에서 다시 설계하는 편이 낫다.
