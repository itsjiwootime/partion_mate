# ADR-20260318 프론트 전역 세션 만료 처리

## 배경
인증 실패 응답이 JSON으로 표준화되더라도, 프론트가 이를 전역에서 처리하지 않으면 화면마다 401이 개별 에러로 노출되고 사용자는 부분적으로 깨진 UI를 보게 된다. 현재 구조는 access token이 `localStorage`에 남아 있으면 로그인 상태처럼 보이기 때문에, 만료 토큰이 남은 채 새로고침했을 때도 보호 API 호출 이후에야 세션 만료를 알 수 있었다. 리프레시 토큰 도입 전에 access token 만료를 일관되게 감지하고 로그인 복귀 경로를 유지하는 전역 처리 레이어가 필요했다.

## 결정사항
프론트 API 레이어에 전역 401 처리 훅을 추가하고, `AuthContext`가 이를 구독해 세션 정리와 세션 만료 상태 기록을 담당하도록 구성한다. 보호 API가 401을 반환하면 `AuthContext`는 토큰과 프로필 캐시를 정리하고 현재 경로를 `returnTo`로 저장한다. 라우터 내부의 `SessionExpiryHandler` 컴포넌트가 이 상태를 감지해 로그인 화면으로 이동시키고, 로그인 화면에는 토스트와 함께 세션 만료 안내 문구를 노출한다. 로그인/회원가입 API는 전역 세션 만료 처리에서 제외한다.

## 대안
각 화면에서 401을 개별 처리한다.
- 장점: 구현 위치가 명시적이고 화면별 예외 처리에 유연하다.
- 단점: 페이지마다 중복 코드가 생기고, 일부 화면에서 빠뜨리면 세션 만료 UX가 다시 깨진다.

API 레이어에서 `window.location`으로 바로 로그인 페이지로 이동한다.
- 장점: 구현이 단순하고 라우터 외부에서도 강제로 이동시킬 수 있다.
- 단점: SPA 상태와 토스트/복귀 경로 관리가 어려워지고, React 상태와 라우터가 분리된다.

세션 만료 시 홈으로만 보내고 복귀 경로는 유지하지 않는다.
- 장점: 구현이 더 단순하다.
- 단점: 참여, 채팅, 내 파티처럼 인증이 필요한 흐름에서 사용자의 맥락이 끊긴다.

## 결과
긍정적 측면:
- 보호 API에서 401이 발생해도 프론트가 한 곳에서 세션 만료를 정리할 수 있다.
- 로그인 후 원래 화면으로 복귀하는 흐름이 유지된다.
- 이후 refresh token 재발급 로직을 같은 API 레이어에 확장하기 쉬워진다.

부정적 측면:
- `AuthContext`, API 레이어, 라우터 사이에 인증 실패 상태 전달 규약이 추가된다.
- 프론트 테스트 러너가 아직 없어 이 카드의 자동 검증은 현재 `npm run build` 수준에 머문다.

중립적 측면:
- 공개 화면에서 선택적으로 호출하는 보호 API(`getMe`)도 401이면 세션 만료 처리 대상으로 간주한다. 이는 공개 화면에서도 로그인 상태 정합성을 유지하는 대신, 일부 경우 로그인 화면으로 더 빨리 이동시킬 수 있다.
