# ADR-20260315-store-query-index-strategy

## 배경
- Epic 2의 조회 쿼리는 `store(latitude, longitude)` 범위 조건과 `party(store_id, party_status)` 조건을 핵심 필터로 사용한다.
- 쿼리화만으로는 충분하지 않고, 조회 패턴에 맞는 인덱스가 없으면 데이터가 늘어날수록 풀스캔 비용이 다시 커질 수 있다.
- 로드맵상 E2-3은 인덱스 추가와 `EXPLAIN` 기반 검증까지 남겨야 완료로 볼 수 있었다.

## 결정사항
- `store` 테이블에 `idx_store_latitude_longitude(latitude, longitude)` 복합 인덱스를 추가했다.
- `party` 테이블에 `idx_party_store_status(store_id, party_status)` 복합 인덱스를 추가했다.
- 검증은 H2 테스트 프로필에서 `INFORMATION_SCHEMA.INDEXES`와 `EXPLAIN` 결과를 함께 확인하는 방식으로 남긴다.

## 대안
- 단일 컬럼 인덱스만 추가
  - 장점: 쓰기 비용이 상대적으로 낮고 단순하다.
  - 단점: 현재 조회 패턴이 복합 조건 중심이라 optimizer가 원하는 만큼 효율적으로 쓰지 못할 수 있다.
- 인덱스 없이 캐시에만 의존
  - 장점: 쓰기 비용이 없다.
  - 단점: cold path 성능과 캐시 miss 상황의 응답시간 개선이 어렵다.
- 공간 인덱스 또는 PostGIS 전환
  - 장점: 위치 기반 조회에 더 적합하다.
  - 단점: 현재 스택 유지라는 프로젝트 기본값을 벗어나고, 운영 복잡도가 커진다.

## 결과
- 긍정적
  - 위치 범위 조회와 지점별 모집 상태 조회 모두 현재 쿼리 패턴에 맞는 복합 인덱스를 가지게 되었다.
  - `EXPLAIN` 결과에 각 인덱스가 명시되어 cold path 최적화 근거를 문서화할 수 있게 되었다.
- 부정적
  - 파티 생성/참여/상태 변경 시 인덱스 유지 비용이 추가된다.
  - 향후 조회 패턴이 바뀌면 현재 복합 인덱스 구성이 최적이 아닐 수 있다.
- 중립적
  - H2 `EXPLAIN`은 MySQL 실행 계획과 완전히 같지는 않지만, 테스트 환경에서 인덱스 적용 여부를 빠르게 검증하는 기준으로는 충분하다.
