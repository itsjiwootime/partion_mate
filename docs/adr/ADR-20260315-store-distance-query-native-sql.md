# ADR-20260315-store-distance-query-native-sql

## 배경
- 주변 지점 조회는 `storeRepository.findAll()` 이후 애플리케이션 메모리에서 거리 계산과 정렬을 수행했고, 각 지점의 모집 중 파티 수를 별도 쿼리로 집계했다.
- 이 구조는 지점 수가 늘어날수록 `전체 지점 적재 + 자바 정렬 + 지점별 카운트 조회`가 겹쳐져 Epic 2의 핵심 병목이 되었다.
- 같은 시점에 지점별 파티 조회도 엔티티 전체 적재 후 DTO 변환을 수행하고 있어, 조회 전용 경로를 더 가볍게 만들 필요가 있었다.

## 결정사항
- 주변 지점 조회는 native SQL projection으로 옮기고, DB에서 거리 계산과 모집 중 파티 수 집계를 함께 수행한다.
- 1차 조회는 반경 50km `bounding box`로 후보를 줄이고, 결과가 없을 때만 전국 거리순 fallback 쿼리를 실행한다.
- 지점별 파티 조회는 엔티티 적재 대신 `PartyResponse` projection 쿼리로 바꾸고, 현재 참여 수량은 `sum(partymember.requestedQuantity)`로 집계한다.
- 서비스 레이어는 projection 결과만 DTO로 매핑하고, 거리 계산 로직은 더 이상 애플리케이션 메모리에서 수행하지 않는다.

## 대안
- 기존 자바 계산 유지
  - 장점: 구현이 단순하고 DB 함수 의존이 없다.
  - 단점: 전체 지점 적재와 N+1 카운트가 계속 남아 조회량 증가에 취약하다.
- JPQL constructor query 사용
  - 장점: 엔티티 중심으로 유지할 수 있고 DB 종속성이 줄어든다.
  - 단점: 삼각함수 조합의 타입 추론이 불안정했고, Hibernate 파서 검증 단계에서 쿼리 생성이 실패했다.
- PostgreSQL/PostGIS 전환
  - 장점: 공간 조회와 인덱싱을 더 정교하게 다룰 수 있다.
  - 단점: 현재 프로젝트 기본 스택과 로드맵 기본값이 MySQL 유지라 범위를 벗어난다.

## 결과
- 긍정적
  - 주변 지점 조회가 DB 주도 경로로 바뀌면서 전체 지점 적재와 지점별 카운트 N+1이 제거됐다.
  - bounding box + fallback 구조로 일반 케이스 성능을 개선하면서도 결과가 아예 비는 UX를 피할 수 있다.
  - 지점별 파티 조회도 응답에 필요한 필드만 projection으로 읽게 되어 조회 전용 경로가 명확해졌다.
- 부정적
  - 거리 계산식과 SQL 함수 사용으로 인해 DB 함수 호환성에 더 민감해졌다.
  - native query는 JPQL보다 타입 안정성이 낮고, 컬럼 alias와 DTO 매핑 규칙을 계속 맞춰야 한다.
- 중립적
  - 반경 50km bounding box는 서비스 성격상 합리적이지만, 실제 서비스 정책이 바뀌면 값 조정이나 검색 전략 재설계가 필요하다.
  - fallback 쿼리가 있어 극단적인 무결과 상황은 완화되지만, 전국 단위 fallback은 여전히 상대적으로 비싼 조회다.
