# ADR-20260316-party-lifecycle-model

- 날짜: 2026-03-16
- 상태: Accepted
- 관련 작업: `E3-1`
- 관련 파일: `src/main/java/com/project/partition_mate/domain/Party.java`, `src/main/java/com/project/partition_mate/domain/PartyStatus.java`, `src/main/java/com/project/partition_mate/dto/PartyResponse.java`

## 배경
- 기존 파티 모델은 `RECRUITING`, `FULL` 상태만 표현할 수 있어서 정원은 찼지만 아직 종료되지 않은 상태와 마감 시간이 지나 완전히 종료된 상태를 구분할 수 없었다.
- 파티 생성 화면에는 날짜/시간 입력이 있었지만 서버 도메인 모델에 마감 시간이 없어서 자동 종료, 마감 알림, 종료 이력 표시를 일관되게 구현할 수 없었다.
- 내 파티, 상세 화면, 알림 흐름에서 종료 사유와 종료 시각을 함께 보여주려면 도메인과 응답 DTO가 같은 생명주기 개념을 공유해야 했다.

## 결정사항
- `Party`에 `deadline`, `closedAt`, `closeReason` 필드를 추가하고 `PartyStatus`에 `CLOSED`를 도입한다.
- 파티 생성 시 사용자가 마감 시간을 보내지 않으면 서버에서 `현재 시각 + 1일`을 기본값으로 채운다.
- 정원 달성은 계속 `FULL`로 표현하고, 시간 만료 또는 후속 종료 정책으로 완전히 닫힌 경우에만 `CLOSED`로 전이한다.
- 파티 응답 DTO와 프론트 정규화 유틸에 종료 필드를 함께 노출해 목록, 상세, 내 파티 화면이 같은 기준으로 마감 상태를 렌더링하도록 맞춘다.

## 대안
- `FULL` 상태만 재사용하고 `deadline`만 추가
  - 장점: 상태 값이 늘지 않아 기존 코드 수정 범위가 작다.
  - 단점: 정원 마감과 시간 종료를 구분할 수 없어 알림, 취소 제한, 화면 표현이 모호해진다.
- 종료 이력을 별도 엔티티로 분리
  - 장점: 종료 사유 변천이나 감사 로그를 더 풍부하게 저장할 수 있다.
  - 단점: 현재 단계에서는 조회 경로가 불필요하게 복잡해지고 CRUD보다 생명주기 정합성이 더 중요한 범위를 넘는다.
- 클라이언트에서 현재 시간과 `deadline`만 비교해 종료 여부를 계산
  - 장점: 서버 상태 전이 코드가 줄어든다.
  - 단점: 서버와 클라이언트의 판정이 달라질 수 있고, 마감 알림과 대기열 만료를 트랜잭션 안에서 묶을 수 없다.

## 결과
- 긍정적
  - 파티 도메인이 모집 중, 정원 마감, 종료를 명확히 구분하게 되어 이후 자동 마감과 알림 파이프라인이 단순해졌다.
  - 종료 시각과 종료 사유가 응답에 포함돼 화면에서 일관된 배지와 안내 문구를 렌더링할 수 있게 됐다.
  - `PartyLifecycleIntegrationTest`로 정원 달성, 시간 종료 상태 전이를 함께 검증할 수 있게 됐다.
- 부정적
  - 기존 DTO, 프론트 상태 정규화, 파티 생성 요청까지 수정 범위가 넓어졌다.
  - 생성자에서 마감 시간 검증을 수행하므로 테스트나 스크립트에서 과거 시간을 넣으려면 별도 조정이 필요하다.
- 중립적
  - 현재 `closeReason`은 `DEADLINE_EXPIRED`만 지원하지만, 이후 호스트 취소나 운영자 종료가 필요하면 enum 확장만으로 대응할 수 있다.
