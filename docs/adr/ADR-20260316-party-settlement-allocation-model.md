# ADR-20260316-party-settlement-allocation-model

- 날짜: 2026-03-16
- 상태: Accepted
- 관련 작업: `E6-1`
- 관련 파일: `src/main/java/com/project/partition_mate/domain/PartyMember.java`, `src/main/java/com/project/partition_mate/service/PartyService.java`, `src/main/java/com/project/partition_mate/dto/PartyDetailResponse.java`

## 배경
- 에픽 5에서 예상 총액과 실구매 총액을 분리했지만, 실제로 참여자별 분담 금액을 확정하는 구조는 아직 없었다.
- 소분 공동구매는 참여 수량이 서로 다르기 때문에 단순 `총액 / 인원수`로는 정산할 수 없고, 1원 단위 나머지가 생기는 경우도 안정적으로 처리해야 했다.
- 이후 송금 상태, 픽업, 거래 완료 로직은 모두 참여자별 확정 금액을 기준으로 동작해야 하므로, 정산 확정 시점의 스냅샷을 보존할 필요가 있었다.

## 결정사항
- 참여자별 금액은 별도 정산 엔티티 대신 `PartyMember.expectedAmount`, `PartyMember.actualAmount`에 저장한다.
- 호스트가 `PUT /party/{id}/settlement`로 실구매 총액을 확정하면, 현재 참여 중인 멤버 목록을 기준으로 예상 금액과 확정 금액을 한 번에 계산해 저장한다.
- 금액 배분은 `총액 / 총 요청 수량`의 기본 단가를 먼저 계산하고, 나머지 1원은 요청 수량이 큰 참여자부터 우선 배분한다. 동률일 때는 `memberId` 오름차순을 사용해 결정 순서를 고정한다.
- 정산 결과는 `PartyDetailResponse.settlementMembers`와 `MyJoinedPartyResponse`에 노출해 호스트와 참여자가 같은 계산 결과를 공유하도록 한다.

## 대안
- 별도 `party_settlement` / `member_settlement` 엔티티를 추가
  - 장점: 정산 이력과 버전 관리를 더 풍부하게 남길 수 있다.
  - 단점: 현재 프로젝트 규모에서 테이블과 조인 복잡도만 증가하고, 에픽 6의 핵심은 정산 상태 전이보다 참여자 운영 흐름이다.
- 금액을 요청 시점마다 계산만 하고 저장하지 않음
  - 장점: 스키마 변경이 적고 구현이 단순하다.
  - 단점: 실구매가가 확정된 뒤에도 계산 기준이 계속 흔들리고, 이후 송금/후기 상태와 결합하기 어렵다.
- 나머지 1원을 무조건 호스트에게 몰아줌
  - 장점: 구현이 단순하다.
  - 단점: 참여자 입장에서 분담 기준이 불투명하고, 호스트 요청 수량이 0인 경우 정책이 어색해진다.

## 결과
- 긍정적
  - 정산 확정 시점의 참여자별 금액이 스냅샷으로 남아 이후 송금/거래 완료 처리 기준이 명확해졌다.
  - 나머지 배분 정책이 결정적이라 테스트와 설명이 쉬워졌다.
  - 호스트 상세 화면과 참여자 내 파티 화면이 같은 정산 결과를 사용하게 됐다.
- 부정적
  - 재정산이 필요한 경우 기존 송금 상태와 충돌할 수 있어, 현재는 송금이 진행된 뒤 재확정을 제한한다.
  - 정산 필드가 `PartyMember`에 추가되면서 참여 모델 책임이 늘었다.
- 중립적
  - 향후 정산 이력이나 취소/재정산 버전 관리가 필요해지면 별도 정산 엔티티로 분리할 여지는 남겨뒀다.
