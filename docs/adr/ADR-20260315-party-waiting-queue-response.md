# ADR-20260315 Party Waiting Queue Response

- 상태: Superseded
- 대체 ADR: `ADR-20260327-remove-waiting-queue-join-fail-fast.md`

## 배경
정원 초과 요청을 단순 실패로 처리하면 사용자 입장에서 같은 파티를 반복 조회하고 다시 시도해야 한다. 로드맵 E1-3의 목표는 정원 초과 요청을 대기열로 전환하고, 클라이언트가 즉시 대기 상태를 이해할 수 있는 응답 모델을 제공하는 것이다.

## 결정사항
`joinParty`는 즉시 참여 가능한 경우 `JOINED`, 잔여 수량이 부족하거나 이미 `FULL` 상태인 경우 `WAITING`을 반환하도록 분기한다. API 응답은 `JoinPartyResponse`로 통일하고 `joinStatus`, `message`, `waitingPosition`, `party`를 포함한다. HTTP 상태는 즉시 참여 시 `201 Created`, 대기열 등록 시 `202 Accepted`를 사용한다.

## 대안
대안 1은 기존처럼 `409 CONFLICT`로 실패시키는 방식이었다. 구현은 가장 단순하지만, 대기열 자동 승격 흐름과 사용자 경험을 만들 수 없고 이후 E1-4와도 맞지 않는다.

대안 2는 대기열에 등록하더라도 기존 `PartyResponse`만 반환하는 방식이었다. 클라이언트가 성공 참여와 대기 등록을 구분할 수 없어 화면/토스트에서 분기하기 어렵다.

대안 3은 별도 `/waiting-queue` API를 두고 참여 API는 실패시키는 방식이었다. 책임 분리는 되지만 사용자가 한 번 더 요청해야 하고, 같은 사용자 액션이 두 API로 쪼개져 UX와 서버 코드가 모두 복잡해진다.

## 결과
긍정적으로는 참여 요청 한 번으로 즉시 참여와 대기 등록을 모두 처리할 수 있고, 클라이언트는 `joinStatus`와 `waitingPosition`으로 명확히 분기할 수 있다. 이후 내 파티 목록과 자동 승격 로직도 같은 상태 모델을 재사용할 수 있다.

부정적으로는 기존 참여 API 응답 타입이 바뀌어 프론트와 테스트를 함께 수정해야 한다. 또한 `202 Accepted`를 채택한 만큼 클라이언트에서 일반 성공과 대기 성공을 구분해 처리해야 한다.

중립적으로는 현재 응답은 대기열 위치만 알려주고 예상 승격 시점 같은 고급 정보는 제공하지 않는다. 이는 현재 범위를 지키기 위한 결정이며, 추후 운영 데이터가 쌓이면 별도 확장이 가능하다.
