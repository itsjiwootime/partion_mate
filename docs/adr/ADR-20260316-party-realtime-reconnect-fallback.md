# ADR-20260316-party-realtime-reconnect-fallback

- 날짜: 2026-03-16
- 상태: Accepted
- 관련 작업: `E4-4`
- 관련 파일: `frontend/src/utils/partyRealtime.js`, `frontend/src/pages/PartyList.jsx`, `frontend/src/pages/PartyDetail.jsx`, `frontend/src/pages/MyParties.jsx`, `docs/specs/E4-party-realtime-events.md`

## 배경
- SSE는 브라우저나 네트워크 상태에 따라 연결이 끊길 수 있으므로, 단순 연결만으로는 최종 상태 일관성을 보장할 수 없다.
- 현재 구현은 `Last-Event-ID` 재전송을 지원하지 않기 때문에, 끊긴 동안 발생한 이벤트는 별도 보정 경로가 필요하다.
- 프론트 테스트 러너가 아직 없어 자동 테스트 범위가 제한적이므로, 재연결 규칙을 코드와 문서로 명확히 남겨야 했다.

## 결정사항
- 클라이언트는 SSE 오류 시 현재 연결을 닫고 최대 5초까지 선형 backoff로 재연결한다.
- 오류가 발생하는 즉시 각 화면의 기존 조회 API를 한 번 다시 호출해 누락 가능성을 보정한다.
- 재연결 성공은 `connected` 이벤트로 판정하고, 화면 상태를 `reconnecting -> live`로 전환한다.
- 자동 검증은 백엔드 SSE 통합 테스트와 프론트 빌드 검증까지로 두고, 재연결/fallback 정책의 세부 동작은 스펙 문서에 명시한다.

## 대안
- `EventSource` 기본 재연결만 그대로 사용
  - 장점: 구현이 가장 단순하다.
  - 단점: 화면이 끊김 상태를 알기 어렵고, 누락 이벤트 보정 타이밍이 없다.
- 지수 backoff와 재시도 제한 도입
  - 장점: 장기 장애 시 재시도 부하를 더 잘 제어할 수 있다.
  - 단점: 현재 범위에서는 정책이 과하고 복잡성이 늘어난다.
- `Last-Event-ID`와 서버 측 이벤트 replay 지원
  - 장점: 누락 복구가 가장 정확하다.
  - 단점: 현재 메모리 기반 emitter 구조와 범위를 넘는 구현이 필요하다.

## 결과
- 긍정적
  - 연결 장애가 발생해도 화면이 기존 API 재조회로 최종 상태를 회복할 수 있게 됐다.
  - 사용자는 실시간 연결 상태를 바로 인지할 수 있다.
  - 브라우저 기본 SSE와 현재 API를 조합해 추가 인프라 없이 fallback 전략을 구현했다.
- 부정적
  - 재연결 중에는 실시간 대신 재조회에 의존하므로 순간적인 지연이 생길 수 있다.
  - 자동 테스트가 프론트 재연결 시나리오까지 완전히 커버하지는 못한다.
- 중립적
  - 이후 프론트 테스트 러너를 도입하면 재연결 유틸과 fallback 호출을 별도 자동화 테스트로 확장할 수 있다.
