# ADR-20260316-notification-ux-exposure

- 날짜: 2026-03-16
- 상태: Accepted
- 관련 작업: `E3-5`
- 관련 파일: `src/main/java/com/project/partition_mate/controller/UserController.java`, `src/main/java/com/project/partition_mate/service/UserService.java`, `frontend/src/pages/Notifications.jsx`, `frontend/src/components/BottomNav.jsx`

## 배경
- 참여 완료, 대기열 승격, 파티 종료 알림이 쌓여도 사용자가 확인할 화면과 API가 없으면 기능 가치가 거의 없다.
- 내 파티 화면에만 메시지를 섞어 넣으면 알림 타입이 늘어날수록 맥락이 섞이고, 후속 채팅/실시간 기능과 역할이 충돌할 수 있다.
- 현재 범위에서는 읽음 처리나 세분화 필터보다, 관련 파티로 다시 이동할 수 있는 단순한 알림 히스토리 화면이 우선이었다.

## 결정사항
- 인증 사용자 전용 `GET /api/users/me/notifications` API를 추가하고 생성 시각 역순으로 앱 내 알림을 조회한다.
- 프론트에는 `/notifications` 전용 화면과 하단 네비게이션 진입점을 추가한다.
- 알림 응답에는 `title`, `message`, `linkUrl`, `createdAtLabel`을 포함해 바로 읽고 관련 파티 상세로 이동할 수 있게 한다.
- 비로그인 상태에서는 알림 내용을 노출하지 않고 로그인 유도 메시지만 보여준다.

## 대안
- 내 파티 화면에 알림 섹션을 같이 표시
  - 장점: 새 화면 없이 구현할 수 있다.
  - 단점: 파티 목록과 알림 히스토리의 관심사가 섞이고 알림이 많아질수록 화면 복잡도가 커진다.
- 서버에서 웹소켓 푸시만 제공하고 별도 이력은 저장하지 않음
  - 장점: 구현이 단순하고 실시간 느낌을 줄 수 있다.
  - 단점: 사용자가 나중에 다시 확인할 수 있는 기록이 남지 않는다.
- 읽음 처리와 필터까지 한 번에 구현
  - 장점: 완성도 높은 알림 센터가 된다.
  - 단점: Epic 3 범위를 넘는 상태 관리와 UI 복잡도가 생긴다.

## 결과
- 긍정적
  - 알림 기능이 단순 백엔드 저장에 그치지 않고 실제 사용자 확인 흐름까지 닫혔다.
  - 참여 완료, 승격, 종료 이벤트를 한 화면에서 모아 볼 수 있어 실사용성이 올라갔다.
  - `UserNotificationControllerTest`와 프론트 빌드로 API/화면 경로를 함께 검증할 수 있다.
- 부정적
  - 읽음 처리, 배지 카운트, 유형별 필터는 아직 없어 알림이 많아지면 사용성이 떨어질 수 있다.
  - 현재 링크는 관련 파티 이동만 지원하므로 세부 액션은 제한적이다.
- 중립적
  - 전용 알림 화면을 먼저 만든 덕분에 이후 읽음 처리나 실시간 뱃지 추가는 같은 경로를 확장하는 방향으로 진행할 수 있다.
