# ADR-20260318-notification-and-settlement-settings-scope

- 날짜: 2026-03-18
- 상태: Accepted
- 관련 작업: `E18-3`
- 관련 파일: `frontend/src/pages/Profile.jsx`, `src/main/java/com/project/partition_mate/controller/UserController.java`, `src/main/java/com/project/partition_mate/service/UserService.java`

## 배경
- 브라우저 푸시 설정은 타입별 토글만 있어 여러 알림을 한 번에 켜거나 끄는 사용성이 부족했다.
- 또한 사용자가 정산이나 송금 공지에 반복해서 쓰는 문구를 저장할 곳이 없어, 이후 운영 흐름에서 재사용할 개인 기본값이 비어 있었다.
- 이미 알림 선호도와 프로필 수정 API가 나뉘어 있는 구조라, 이번 카드에서는 기존 계약을 깨지 않으면서 설정 화면만 확장하는 방식이 필요했다.

## 결정사항
- 알림 설정은 기존 `notification-preferences` API를 그대로 재사용하고, 프론트에서 지원 가능한 알림 타입 전체를 한 번에 토글하는 UI만 추가한다.
- 정산 기본 안내는 사용자별 단일 텍스트 설정으로 저장하고, `GET/PUT /api/users/me/settlement-settings` 엔드포인트로 조회/수정한다.
- 정산 기본 안내는 `User`에 nullable 필드로 저장하고, 공백 입력은 `null`로 정리한다.
- 이번 카드에서는 브라우저 푸시와 정산 기본 안내까지만 다루고, 앱 내 알림 자체 비활성화나 정산 문구 자동 삽입은 범위 밖으로 둔다.

## 대안
- 알림과 정산 설정을 모두 하나의 `user_settings` 엔티티와 단일 API로 새로 묶는 방식
  - 장점: 장기적으로 사용자 설정 도메인을 한곳에 모을 수 있다.
  - 단점: 현재 존재하는 알림 선호도 구조와 중복되어 이번 카드 범위에 비해 변경량이 커진다.
- 정산 기본 안내를 프로필 수정 API에 포함시키는 방식
  - 장점: 엔드포인트 수가 줄고 구현이 단순하다.
  - 단점: 프로필 수정과 운영 설정의 책임이 섞이고, `E18-1`에서 정한 계약을 다시 넓혀야 한다.
- 알림 전체 토글 없이 개별 토글만 유지하는 방식
  - 장점: 백엔드/프론트 로직이 가장 단순하다.
  - 단점: 지원 알림을 한 번에 켜고 끄는 기본적인 설정 UX가 부족하다.

## 결과
- 긍정적 트레이드오프: 사용자는 지원 알림을 한 번에 토글할 수 있고, 정산 공지 기본값을 프로필 설정 화면에서 관리할 수 있다.
- 부정적 트레이드오프: 정산 기본 안내는 아직 실제 파티 운영 화면에 자동 주입되지 않으므로 저장 즉시 체감되는 범위는 제한적이다.
- 중립적 트레이드오프 또는 후속 영향 사항: 향후 정산 기본 안내를 채팅 공지나 정산 확정 폼과 연결할 때는 이번 설정 필드를 재사용하면 된다.
