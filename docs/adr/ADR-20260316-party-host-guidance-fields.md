# ADR-20260316-party-host-guidance-fields

- 날짜: 2026-03-16
- 상태: Accepted
- 관련 작업: `E5-2`
- 관련 파일: `src/main/java/com/project/partition_mate/domain/Party.java`, `frontend/src/pages/CreateParty.jsx`, `frontend/src/pages/PartyDetail.jsx`, `frontend/src/utils/party.js`

## 배경
- 실제 소분 모임에서는 호스트가 봉투를 준비하는지, 현장 소분인지, 보냉백이 필요한지 같은 운영 정보가 참여 결정에 직접 영향을 준다.
- 기존 화면에는 오픈채팅 링크 외에 거래 방식 자체를 구조적으로 설명할 수단이 부족해서, 참여자가 파티 상세를 봐도 준비물을 충분히 파악하기 어려웠다.
- 이 정보가 텍스트 메모에만 섞이면 목록/상세/후속 정산 흐름에서 재사용하기 어렵고, 화면별 표현도 제각각이 된다.

## 결정사항
- `Party`에 `hostProvidesPackaging`, `onSiteSplit`, `guideNote`를 추가해 호스트 준비 상태와 거래 안내를 구조화한다.
- 생성 화면에는 체크박스와 안내 메모 입력란을 넣고, 상세 화면에는 `소분 정보`, `거래 안내`, `가격 정보` 섹션을 분리해 거래 조건을 읽기 쉽게 노출한다.
- `guideNote`는 자유 텍스트를 허용하되, 반복적으로 판단이 필요한 항목은 boolean 필드로 분리해 API/화면 로직에서 직접 사용할 수 있게 한다.
- 프론트 정규화 유틸과 카드 컴포넌트까지 같은 필드를 공유하도록 맞춰, 상세 진입 경로가 달라도 동일한 정보를 표시한다.

## 대안
- 모든 거래 안내를 `guideNote` 하나의 자유 텍스트로만 저장
  - 장점: 구현이 단순하고 유연하다.
  - 단점: 자주 쓰는 조건을 화면에서 구조적으로 강조할 수 없고, 후속 필터나 요약 배지로 활용하기 어렵다.
- 준비물 관련 정보를 별도 `PartyNotice` 엔티티로 분리
  - 장점: 공지 이력 관리나 여러 공지 확장에 유리하다.
  - 단점: 현재 범위에서는 파티 단일 메타데이터로 충분하며, 별도 엔티티는 CRUD와 조회 비용만 늘린다.
- 오픈채팅으로만 안내하도록 서버 모델은 변경하지 않음
  - 장점: 개발 비용이 가장 적다.
  - 단점: 서비스 안에서 거래 조건을 확인할 수 없어 파티 상세의 완성도가 낮고, 채팅이 열리기 전까지 핵심 정보를 알기 어렵다.

## 결과
- 긍정적
  - 참여자는 파티 상세만 보고도 포장 제공 여부, 현장 소분 여부, 추가 준비 사항을 빠르게 이해할 수 있게 됐다.
  - 생성 폼과 상세 화면이 같은 메타데이터를 써서 UX와 API가 일관된다.
  - 이후 픽업/정산/채팅 기능에서도 같은 거래 안내 데이터를 재사용할 수 있다.
- 부정적
  - 생성 폼 입력 항목이 늘어나 초기 작성 부담이 조금 커졌다.
  - boolean 필드만으로 모든 소분 상황을 표현할 수는 없어, 일부 복잡한 조건은 여전히 안내 메모 의존이 남는다.
- 중립적
  - 향후 호스트 공지, 픽업 공지와 연결되면 `guideNote`를 공지 스냅샷으로 확장할 수 있지만 현재는 파티 생성 시점 정보로 한정한다.
