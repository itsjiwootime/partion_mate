# ADR-20260318-chat-safety-settings-test-scope

- 날짜: 2026-03-18
- 상태: Accepted
- 관련 작업: `E20-3`
- 관련 파일: `frontend/src/pages/Chat.flow.test.jsx`, `frontend/src/pages/Profile.notificationSettings.test.jsx`, `frontend/src/pages/Notifications.state.test.jsx`

## 배경
- 채팅과 안전 기능은 이미 구현되어 있었지만, 프론트 테스트는 일부 차단/신고 진입점과 딥링크 정도만 검증하고 있었다.
- 그 결과 채팅방 선택 후 상세 로드, 호스트 공지 저장, 현재 브라우저 Web Push 연결/해제, 알림 화면의 로그인 가드와 오류 복구 같은 실제 상호작용 경로는 자동 검증이 비어 있었다.
- 이 영역은 컨텍스트와 브라우저 상태 의존성이 많아, 단순 렌더링 테스트보다 화면 상태 전환과 API 호출 순서를 보는 테스트 범위 정의가 필요했다.

## 결정사항
- `Chat`은 방 선택, 읽음 처리, 호스트 공지 저장을 별도 흐름 테스트로 추가하고, 기존 차단 fallback 테스트와 함께 묶는다.
- `Profile`은 기존 알림 설정 테스트를 확장해 타입별 토글뿐 아니라 현재 브라우저 Web Push 연결/해제까지 검증한다.
- `Notifications`는 딥링크 테스트를 유지하면서 비로그인 가드와 조회 실패 후 재시도 복구를 별도 테스트로 추가한다.
- 실제 브라우저 권한 팝업, 서비스워커 등록, 실시간 STOMP 수신은 mock 밖 범위로 두고, 화면 상호작용 결과와 후속 상태만 프론트 테스트 책임으로 둔다.

## 대안
- 기존 테스트 파일을 그대로 두고 수동 확인만 유지한다.
  - 장점: 테스트 유지보수 비용이 늘지 않는다.
  - 단점: 채팅과 알림 설정처럼 회귀 위험이 높은 상호작용이 자동 검증 밖에 남는다.
- Playwright 같은 브라우저 E2E로만 검증한다.
  - 장점: 권한 팝업과 서비스워커까지 실제 환경에 가깝게 볼 수 있다.
  - 단점: 현재 카드 범위보다 무겁고, 빠른 회귀 피드백이라는 목적과 맞지 않는다.
- 채팅/프로필/알림을 하나의 거대한 통합 테스트로 묶는다.
  - 장점: 사용자의 큰 흐름을 한 번에 볼 수 있다.
  - 단점: 실패 지점 추적이 어렵고, 페이지별 mock 준비가 과도하게 커진다.

## 결과
- 긍정적
  - 채팅방 선택, 공지 저장, 브라우저 연결/해제, 알림 가드와 재시도 같은 상호작용 회귀를 자동으로 잡을 수 있다.
  - 기존 안전 센터와 차단 테스트를 유지하면서 빈 구간만 보강해 테스트 중복을 줄였다.
  - 페이지별 책임에 맞게 테스트를 분리해 실패 원인을 빠르게 좁힐 수 있다.
- 부정적
  - Web Push와 채팅 클라이언트 mock 구성이 복잡해져 테스트 초기 설정 비용이 늘어난다.
  - 실제 브라우저 권한 팝업이나 서비스워커 동작은 여전히 mock 기반 추정에 의존한다.
  - UI 문구 변경에 따라 테스트 셀렉터 보수가 자주 필요할 수 있다.
- 중립적
  - 이번 결정은 프론트 테스트 러너를 유지한 채 상호작용 범위만 확장한 것이다.
  - 이후 실제 브라우저 기반 E2E가 필요해지면, 현재 테스트는 빠른 회귀망으로 유지하고 권한/서비스워커 시나리오만 별도 계층으로 추가할 수 있다.
