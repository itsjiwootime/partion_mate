# ADR-20260318 프론트 테스트 러너로 Vitest와 React Testing Library 도입

## 배경
프론트엔드는 지금까지 `npm run build`와 수동 클릭 테스트만으로 품질을 확인하고 있었다. 그 결과 로그인 복귀, 세션 만료 리다이렉트 같은 사용자 흐름은 구현 후에도 자동 검증이 어려웠고, 회귀를 막기 위해 매번 수동 확인이 필요했다. 인증 보강 작업이 진행되는 시점에는 컴포넌트와 라우터 흐름을 빠르게 검증할 수 있는 프론트 테스트 러너가 필요했다.

## 결정사항
Vite 기반 프론트엔드에 `Vitest + jsdom + React Testing Library` 조합을 도입한다. `vite.config.js`에 테스트 환경을 설정하고, 공통 `setup.js`에서 `jest-dom` 매처와 브라우저 테스트 보조 설정을 로드한다. 첫 검증 대상으로는 로그인 화면의 `returnTo` 복귀와 세션 만료 시 로그인 화면 이동을 테스트한다. E2E 러너는 이번 단계에서 도입하지 않고, 컴포넌트/라우터 레벨 검증을 우선 구축한다.

## 대안
Jest를 별도로 도입한다.
- 장점: 문서와 예제가 많고 생태계가 성숙했다.
- 단점: Vite 환경과 중복 설정이 생기고, 현재 프로젝트 규모에서는 설정 오버헤드가 더 크다.

Playwright나 Cypress만 바로 도입한다.
- 장점: 실제 브라우저 기준으로 로그인/세션 흐름을 가장 현실적으로 검증할 수 있다.
- 단점: 초기 설정 비용이 더 크고, 지금 필요한 컴포넌트/라우터 단위 회귀 방지에는 과하다.

빌드 검증과 수동 테스트만 유지한다.
- 장점: 추가 도구 없이 계속 개발할 수 있다.
- 단점: 세션 만료, 리다이렉트, 토스트 노출 같은 회귀를 자동으로 잡지 못한다.

## 결과
긍정적 측면:
- 프론트 핵심 사용자 흐름을 `npm run test`로 반복 검증할 수 있다.
- Vite 기반 프로젝트와 자연스럽게 맞물려 설정 부담이 작다.
- 이후 refresh token 재발급, 로그아웃, 보호 라우트 테스트를 같은 스택으로 확장하기 쉽다.

부정적 측면:
- jsdom 기반 테스트는 실제 브라우저와 완전히 같지 않아 일부 상호작용은 여전히 수동 또는 E2E 검증이 필요하다.
- 테스트 러너와 setup 파일 유지보수 비용이 추가된다.

중립적 측면:
- 이번 단계는 컴포넌트/라우터 레벨 검증까지만 포함하며, Playwright 같은 브라우저 E2E는 필요 시 다음 단계에서 추가한다.
