# ADR-20260318-party-operation-flow-test-scope

- 날짜: 2026-03-18
- 상태: Accepted
- 관련 작업: `E20-2`
- 관련 파일: `frontend/src/pages/JoinParty.flow.test.jsx`, `frontend/src/pages/PartyDetail.operationFlow.test.jsx`

## 배경
- `JoinParty`와 `PartyDetail`은 파티 운영의 핵심 화면이지만, 기존 프론트 테스트는 관심 저장, 신고/차단, 신뢰 배지 중심으로만 커버되어 있었다.
- 그 결과 즉시 참여와 대기열 등록의 이동 차이, 호스트의 정산/픽업 확정, 참여자의 송금 완료/픽업 확인/호스트 후기 작성 같은 운영 액션은 수동 확인에 의존하고 있었다.
- 특히 `PartyDetail`은 역할에 따라 렌더링되는 액션이 크게 달라, 같은 API라도 호스트와 참여자 경로를 나눠서 보는 테스트 전략이 필요했다.

## 결정사항
- `JoinParty`는 결과에 따라 라우팅이 달라지는 화면이므로 `MemoryRouter` 기반 통합 성격 테스트로 즉시 참여와 대기열 등록을 각각 검증한다.
- `PartyDetail`은 호스트 fixture와 참여자 fixture를 분리해 정산/픽업 확정, 송금 완료, 픽업 확인, 호스트 후기 등록을 역할별 운영 흐름으로 검증한다.
- 기존 안전/신뢰/관심 저장 테스트는 그대로 유지하고, 이번 카드에서는 운영 액션 회귀만 별도 파일로 추가한다.
- 테스트는 버튼 클릭과 API payload, 성공 토스트, 후속 상태 반영까지만 검증하고, 서버 정책 자체는 백엔드 테스트 책임으로 남긴다.

## 대안
- `PartyDetail` 단일 거대 테스트에 호스트와 참여자 시나리오를 모두 넣는다.
  - 장점: 파일 수가 늘지 않고 한 화면의 모든 동작을 한곳에서 볼 수 있다.
  - 단점: fixture가 과도하게 커지고 실패 시 어느 역할 경로가 깨졌는지 파악하기 어려워진다.
- API 호출 payload만 단순 단위 테스트로 검증하고 라우팅은 보지 않는다.
  - 장점: 테스트 작성 비용이 낮고 mock 준비가 단순하다.
  - 단점: 참여 성공과 대기열 등록의 화면 이동 차이처럼 실제 사용자 결과를 놓치게 된다.
- 브라우저 E2E에서 운영 흐름을 한 번에 검증한다.
  - 장점: 더 실제 환경에 가까운 검증이 가능하다.
  - 단점: 현재 카드 범위를 넘어 인프라 부담이 크고, 빠른 회귀 피드백이라는 목적에 비해 비용이 과하다.

## 결과
- 긍정적
  - 참여 성공, 대기열 등록, 정산, 픽업, 후기 작성의 주요 운영 경로를 자동으로 검증할 수 있다.
  - 호스트와 참여자 역할을 분리해 같은 화면의 조건부 UI 회귀를 더 정확히 잡을 수 있다.
  - 안전/신뢰 테스트와 운영 테스트가 분리되어 실패 원인을 좁히기 쉬워진다.
- 부정적
  - 역할별 fixture와 mock 응답을 유지해야 해서 테스트 데이터 관리 비용이 늘어난다.
  - `PartyDetail`의 UI 구조가 바뀌면 테스트 셀렉터 보수가 잦아질 수 있다.
  - 서버 응답 형태가 조금만 바뀌어도 프론트 테스트 fixture를 함께 조정해야 한다.
- 중립적
  - 이번 결정은 프론트 테스트 러너를 바꾸지 않고 E12 기반 범위를 운영 액션까지 확장한 것이다.
  - 이후 `E20-3`에서는 채팅/안전/알림 설정처럼 상호작용이 많은 화면을 같은 방식의 역할 기반 테스트로 확장할 수 있다.
