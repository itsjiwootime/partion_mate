# ADR-20260316-party-review-write-policy

- 날짜: 2026-03-16
- 상태: Accepted
- 관련 작업: `E7-1`
- 관련 파일: `src/main/java/com/project/partition_mate/service/PartyService.java`

## 배경
- 거래가 끝나기 전에도 후기를 남길 수 있으면 실제 거래 품질이 아닌 기대감만 반영된 평점이 누적될 수 있다.
- 호스트와 참여자가 서로 여러 번 후기를 남기면 신뢰도 지표가 쉽게 왜곡되고, 동일 파티에서 감정적인 반복 평가가 생길 수 있다.
- 에픽 6에서 `TradeStatus.COMPLETED`와 `NO_SHOW`가 분리되었기 때문에 후기 작성 조건을 이 상태 전이와 맞물려 정의할 필요가 있었다.

## 결정사항
- 후기 엔티티는 `party`, `reviewer`, `reviewee`를 모두 보관하고, `(party_id, reviewer_id, reviewee_id)` 유니크 제약으로 동일 파티에서 동일 대상에게 1회만 작성 가능하게 한다.
- 참여자는 자신의 `PartyMember.tradeStatus == COMPLETED`일 때만 호스트에게 후기를 작성할 수 있게 한다.
- 호스트는 거래 완료된 참여자에게만 후기를 남길 수 있게 하고, 노쇼 참여자나 자기 자신은 대상에서 제외한다.
- 후기 작성 API는 `POST /party/{id}/reviews`로 두고, 성공 시 최신 `PartyDetailResponse`를 반환해 화면이 즉시 갱신되게 한다.

## 대안
- 대안 1: 파티가 `CLOSED` 상태이기만 하면 누구나 후기 작성 허용
  - 장점: 구현이 단순하고 별도 참여자 상태 확인이 줄어든다.
  - 단점: 실제 픽업/거래 완료 여부와 무관하게 후기를 남길 수 있어 노쇼 참여자나 미거래 상대에 대한 평가가 섞인다.
- 대안 2: 호스트 승인 후에만 후기 작성 허용
  - 장점: 거래가 끝났다는 확실한 확인 절차를 둘 수 있다.
  - 단점: 승인 누락 시 정상 거래 후기까지 막히고, 후기 시스템이 호스트 액션에 과하게 종속된다.
- 대안 3: 후기 전용 별도 엔티티 없이 `PartyMember`에 평점/코멘트 컬럼 추가
  - 장점: 테이블 수가 줄고 조회가 단순해진다.
  - 단점: 호스트와 참여자의 상호 후기, 동일 파티 다대일 리뷰 관계, 작성자/대상 분리가 모두 어려워진다.

## 결과
- 긍정적: 후기 작성 자격이 `COMPLETED` 상태와 일치해 신뢰도 산식의 입력 데이터 품질이 높아진다.
- 긍정적: 유니크 제약과 서비스 검증을 함께 두어 중복 후기를 방지하고 UI에서 재요청이 와도 안정적으로 막을 수 있다.
- 부정적: 후기 제출 시 현재 사용자 역할과 대상 역할을 모두 확인해야 해서 `PartyService` 조립 로직이 복잡해진다.
- 부정적: 동일 파티에서 같은 상대에 대한 수정 기능은 제공하지 않으므로, 잘못 작성한 후기를 즉시 교정하기 어렵다.
- 중립적: 이후 후기 수정/삭제 기능이 필요해지면 현재 유니크 제약은 유지한 채 soft delete 또는 revision 정책을 추가하는 방향이 자연스럽다.
- 중립적: 후기 작성 성공 후 최신 상세 응답을 반환하는 구조는 프론트 동기화에는 유리하지만, 응답 payload가 다소 커진다.
