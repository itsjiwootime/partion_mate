# ADR-20260317-feedback-and-empty-state-policy

- 날짜: 2026-03-17
- 상태: Accepted
- 관련 작업: `E9-6`
- 관련 파일: `frontend/src/pages/Home.jsx`, `frontend/src/pages/MyParties.jsx`, `frontend/src/pages/Notifications.jsx`, `frontend/src/pages/Profile.jsx`, `frontend/src/pages/Chat.jsx`

## 배경
- 주요 화면에서 실패나 빈 데이터가 발생하면 단순 문장만 보이거나 다음 행동이 없어 사용자가 막히는 지점이 있었다.
- 알림, 내 파티, 채팅처럼 반복 방문하는 화면은 "다시 시도", "다른 화면으로 이동" 같은 후속 행동이 특히 중요하다.
- 화면별로 제각각 상태 피드백을 만들면 문구와 CTA 톤이 쉽게 흐트러진다.

## 결정사항
- 로딩은 기존 `LoadingState`를 재사용하고, 빈 상태는 `EmptyState` 기반으로 통일한다.
- 오류 상태에는 가능한 한 같은 화면을 다시 시도할 수 있는 재시도 버튼을 추가한다.
- 빈 상태에는 사용자가 다음으로 갈 수 있는 가장 가까운 행동을 CTA로 연결한다. 예를 들어 파티가 없으면 `/parties`, 채팅방이 없으면 파티 탐색으로 유도한다.
- 위치 기반 홈 화면처럼 컨텍스트가 뚜렷한 경우에는 빈 상태 설명에 원인과 재시도 방향을 함께 적는다.

## 대안
- 화면마다 자유 형식의 로딩/에러/빈 상태 유지
  - 장점: 개별 화면 최적화가 쉽다.
  - 단점: 문구와 CTA 품질이 들쭉날쭉해진다.
- 전역 에러 토스트만 사용하고 화면 내부 피드백은 최소화
  - 장점: 구현이 단순하다.
  - 단점: 사용자가 현재 화면에서 무엇을 해야 할지 알기 어렵다.
- 모든 상태 피드백을 새 공통 컴포넌트로 추상화
  - 장점: 재사용성이 높다.
  - 단점: 현재 범위에서는 과하고, 세부 문구 차이를 담기 어려울 수 있다.

## 결과
- 긍정적
  - 빈 화면과 오류 화면에서도 사용자가 다음 행동을 바로 선택할 수 있게 됐다.
  - `LoadingState`, `EmptyState`, 재시도 버튼 조합으로 주요 화면의 피드백 톤이 정리됐다.
  - 복귀 경로, 탐색 CTA 개선과 맞물려 사용성 저하 지점을 줄였다.
- 부정적
  - 화면 안에 버튼과 설명이 늘어 레이아웃이 조금 더 길어졌다.
  - 아직 모든 화면에 완전히 같은 패턴이 적용된 것은 아니어서 후속 정리가 남아 있다.
- 중립적
  - 추후 프론트 테스트 러너가 들어오면 상태 피드백 렌더링을 컴포넌트 단위로 자동 검증하는 것이 자연스럽다.
