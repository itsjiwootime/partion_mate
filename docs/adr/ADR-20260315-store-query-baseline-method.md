# ADR-20260315 Store Query Baseline Method

## 배경
Epic 2의 목표는 위치 기반 조회 최적화 전후의 성능 차이를 수치로 남기는 것이다. 최적화를 시작하기 전에 현재 구현의 기준값을 반복 가능하게 측정할 방법과 데이터셋 규모를 먼저 정해야 했다.

## 결정사항
E2-1의 기준 측정은 `StoreQueryBaselineBenchmarkTest`와 `scripts/run_store_query_baseline.sh`로 수행한다. 측정 대상은 `GET /api/stores/nearby`와 `GET /api/stores/{storeId}/parties` 두 조회 API이며, 데이터셋은 지점 300개와 지점당 파티 6개로 고정한다. 각 API는 워밍업 5회 후 측정 20회로 평균 응답시간과 P95를 기록한다.

## 대안
대안 1은 서비스 메서드만 직접 호출해 측정하는 방식이었다. 구현은 더 단순하지만, 컨트롤러와 직렬화 비용이 빠져 실제 API 기준값으로 보기 어렵다.

대안 2는 k6 같은 외부 HTTP 도구를 바로 도입하는 방식이었다. 실운영에 더 가까운 측정이 가능하지만, 현재 단계에서는 별도 환경 준비와 시드 주입 흐름이 늘어나 기준값 확보보다 설정 비용이 더 커진다.

대안 3은 매우 작은 데이터셋으로 빠르게 측정하는 방식이었다. 실행은 빠르지만, 현재 `findAll + 거리 계산 + 지점별 카운트` 구조의 비용이 충분히 드러나지 않아 이후 최적화 효과를 비교하기 어렵다.

## 결과
긍정적으로는 최적화 전 현재 조회 구조의 평균 응답시간과 P95를 저장소 안에서 반복 재생산할 수 있게 된다. 또한 같은 데이터셋과 반복 횟수를 유지하므로 E2-2 이후 개선 폭을 직접 비교하기 쉽다.

부정적으로는 H2 + MockMvc 기반이므로 실제 네트워크 오버헤드와 MySQL 인덱스 특성은 반영되지 않는다. 따라서 이 수치는 내부 기준값으로 해석해야 하고, 운영 유사 환경 측정은 후속 과제로 남는다.

중립적으로는 지점 300개, 파티 1800개 데이터셋은 현재 구현 비용을 드러내기엔 충분하지만 절대적인 실서비스 규모를 뜻하지는 않는다. 비교 가능성을 우선한 선택이다.
